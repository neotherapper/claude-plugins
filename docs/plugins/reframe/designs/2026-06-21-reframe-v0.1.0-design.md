# reframe v0.1.0 — Design Spec

**Date:** 2026-06-21
**Status:** Approved design — ready for implementation planning
**Name:** `reframe` (decided 2026-06-22)
**Author:** Georgios Pilitsoglou (with Claude)

---

## 1. Summary

`reframe` is a new Claude Code plugin (sibling to `beacon`) that takes the URL of an
**existing website** and produces a **purpose-driven redesign brief** for Anthropic's
**Claude Design** (`claude.ai/design`), plus the durable strategic analysis behind it.

It reads a site's content and structure, evaluates it against category-specific best
practices, and synthesizes a concise, high-signal brief that — pasted into Claude Design
alongside its native web-capture — drives a redesign that is *better for the site's
purpose*, not just a reskin.

**The job in one line:** Claude Design captures *what a site looks like*; `reframe`
argues *what it should accomplish and why the current one underperforms*.

---

## 2. Problem & motivation

- **Beacon doesn't cover this.** `beacon:site-recon` is technical reconnaissance — API
  surfaces, framework fingerprinting, security exposure. Its `site-map.md` is a raw URL
  list; it never reads what a page is *for*. None of its output feeds a redesign.
- **Claude Design has craft but no strategy.** Opus 4.7 has general design craft baked
  in, and Claude Design natively web-captures a live URL and extracts a design system
  from a codebase/Figma. But web-capture *clones current patterns* — it does not reason
  about whether the current IA/content/positioning serves the site's purpose.
- **The gap is strategic/semantic extraction:** purpose, audience, goals, information
  architecture, content audit, and a critique grounded in category best-practice. Neither
  Beacon nor Claude Design produces it. That gap is the entire reason this plugin exists.

**Evidence the gap is real and the brief must be concise:** Claude Design is token-heavy
(a design-system setup + one prototype can consume ~50% of a Pro weekly allotment; large
inputs degrade performance). So the deliverable is a *tight strategy brief*, never a data
dump.

---

## 3. Non-goals (what reframe deliberately does NOT do)

- **No design-system / token extraction.** Claude Design and `design-extract` already do
  this. reframe stays strictly *upstream* of the visual layer.
- **No API/endpoint/security/PII/OSINT analysis.** That is Beacon's job; none of it serves
  a redesign brief.
- **No greenfield/from-idea brief generation.** Input is always an existing live site.
  (The user's existing hand-written briefs like MyPenFlow/LunaGram are greenfield; reframe
  is the existing-site-redesign counterpart and reuses their *brief shape*, not their flow.)
- **No multiplayer, no rendering the redesign itself.** reframe produces the brief; Claude
  Design produces the design.

---

## 4. Architecture

Standalone plugin, **Approach 1** (own lightweight content pass; Beacon optional):

- A single prompt-driven **skill** (like `beacon:site-recon`), not a compiled program.
- **Reuses Beacon *patterns*, not Beacon's scope:**
  - Running in-context **session brief** appended per phase, flushed to files at the end.
  - **Tool-availability matrix** with WAF fallback — *copied* and trimmed (only 2 skills
    exist; a shared library is premature). Trim the WAF chain to **Firecrawl → Jina →
    browser-fetch**; drop Scrapfly/Spider/DataDome specialists (they serve API probing).
  - **Packs as pluggable domain knowledge** — Beacon has *tech packs*; reframe has
    **category packs** (§8). Bundled **locally**, not via Beacon's remote version-pinned
    GitHub-raw scheme (categories are evergreen and few).
- **Composes with Beacon when present:** if `docs/research/{slug}/tech-stack.md` already
  exists, read only the platform/CMS line for the brief's tech-constraint note (e.g. "it's
  Webflow → respect Webflow CMS limits"). Never *requires* a Beacon run.

---

## 5. Pipeline (coverage-first)

Nine phases. The ordering principle is **render → verify coverage → then infer** — because
the flagship target (`trustyourphysio.com`) is a React SPA that returns *zero* rendered
body text to `curl`/WebFetch. Inferring before verifying coverage would confidently brief
on an empty shell.

| # | Phase | What happens | Writes |
|---|-------|--------------|--------|
| 1 | **Scaffold + tool check** | Slugify domain; create output folder + empty files; detect crawlers (Jina/Firecrawl/WebFetch) **and Chrome DevTools MCP** (render + screenshot is first-class) | — |
| 2 | **Structure discovery** | `sitemap.xml`, `robots.txt`, homepage nav → enumerate **all** URLs; cluster by path template (`/blog/*`, `/services/*`, …) | `ia-map.md` (skeleton) |
| 3 | **Render + coverage gate** | Fetch homepage; if rendered text is near-empty (SPA), escalate to headless render. Apply a **content-sufficiency gate**: if real content is below threshold (placeholder/coming-soon), enter **greenfield mode** (do not infer; tell user). Emit a **coverage manifest** (what was/wasn't reachable; auth/WAF gating) | session brief |
| 4 | **Content crawl + screenshots** | **Token-budgeted** crawl (not page-count): sample **1–2 pages per template cluster**, floor = homepage + nav targets. Capture clean markdown **+ 1 screenshot per sampled template** (above-fold + full-page for homepage) | `.crawl/` |
| 5 | **Content audit (evaluative)** | Per page: type, purpose, value props, CTAs, forms, media **+ a verdict: keep / revise / consolidate / remove** + ROT flags (redundant/outdated/trivial, thin, orphan, off-message), each tied to a goal | `content-inventory.md` |
| 6 | **IA / journey map** | Nav hierarchy + page purposes; **a small journey set** (1 primary + 2–3 secondary, keyed to entry intent) with the decisive objection each must resolve; primary conversion path. **Scales to site size** (≤N pages → section-level anchor map, not a hollow page table) | `ia-map.md` |
| 7 | **Intent inference** | Infer purpose, audience, goals, **category** + per-field confidence. Always ask the **one uninferable question: "redesign for the same purpose, or a new one?"** Record *current purpose (inferred)* vs *target purpose (declared)*. Below-confidence categories note the top-2 hypotheses | session brief |
| 8 | **Current-design critique** | Judge the site against the **loaded category pack**, using **markdown + screenshots** (visual hierarchy, density, dated-ness) + a **voice/messaging** pass + content-side SEO/a11y signals (heading structure, metadata, schema/NAP, alt-text) → keep/fix. Each finding is **severity-rated 0–4 + cites the named best-practice it violates + a concrete fix** (format borrowed from `frontend-design-audit`) | `current-critique.md` |
| 9 | **Synthesize** | Emit the two-stage deliverable (§9): `brief.md` (onboarding) + `run-sheet.md` (canvas prompts). Resolve `INDEX.md` | `brief.md`, `run-sheet.md`, `INDEX.md` |

## 6. Output structure

```
docs/redesign/{site-slug}/
├── INDEX.md              ← summary, assumptions header, coverage manifest, "how to use"
├── brief.md             ← THE Claude Design onboarding brief (the headline deliverable)
├── run-sheet.md         ← sequential canvas prompts (screens/components), derived from real pages
├── content-inventory.md  ← evaluative content audit (keep/revise/consolidate/remove + ROT)
├── ia-map.md            ← information architecture + journeys + conversion path
├── current-critique.md   ← keep/fix vs category best-practice (incl. screenshots referenced)
└── .crawl/              ← raw per-page markdown + screenshots (git-ignored working data)
```

Slug derivation matches Beacon: strip `www.`, then `example.com → example-com`.
`.crawl/` is **git-ignored** (raw HTML dumps and PNGs should not be committed); the five
markdown files + the brief are the committed artifacts.

## 7. The brief — `brief.md` (two-stage deliverable)

The brief mirrors **Claude Design's actual onboarding fields** (a "what it is / audience /
distance-from" block + a "design-system notes" block), because the extracted design system
persists across Claude Design projects and should be separable from per-screen generation.

**`brief.md` sections (the pasteable onboarding block):**

1. **Assumptions header** *(editable, first)* — inferred purpose, audience, category, goal;
   current-vs-target purpose. Printed so a wrong inference is correctable at a glance.
2. **What the site is** — inferred purpose / value proposition.
3. **Target audience** — grounded in on-page evidence (and proxies where available).
4. **Redesign goals & success criteria** — 3–5 *measurable / judgeable* statements
   (e.g. "a first-time visitor understands what this does in 5 seconds"), not adjectives.
5. **KEEP / CHANGE / ADD**:
   - **KEEP** — brand assets (logo, brand color, wordmark — a *constraint*, the one visual
     thing reframe extracts), ranking/high-value URLs **+ a redirect map**, working flows.
   - **CHANGE** — concrete keep/fix deltas from the critique.
   - **ADD** — coverage gaps (journey stages with no supporting content).
6. **Information architecture (proposed)** — the screen/page list (enumerated, inferable
   from the captured IA) + primary journeys. **Each page carries an intent triplet**:
   *concrete subject · target audience · the page's single job* — pre-filled from the
   content audit. (This is exactly the per-page framing `frontend-design` asks a user to
   supply by hand and that Claude Design's handoff bundle expects; reframe hands it over
   for free, modeled on Relume's IA-first input frame reverse-engineered from the site.)
7. **Design direction + design-system seed** — **concrete opinionated seed values** from
   the category pack: 3–5 hex palette with hierarchy, body+UI font candidates, spacing
   base, radius philosophy, motion durations, borders-vs-shadows stance. (Pure inference
   yields "modern/clean/trustworthy" mush; seed values are the only lever on taste.)
8. **Reference + anti-reference sites** + strict NOs (taste in 3 words: "Wirecutter ×
   Kinfolk" does more per token than prose).
9. **Web-capture instruction (critical)** — *"Capture the live URL for content, structure,
   and brand-assets-to-KEEP only. The design direction above OVERRIDES all captured visual
   styling."* Prevents reproducing the site being replaced.
10. **Tech / export target + handoff** — stack note (from Beacon if present) + "export ZIP
    / Open in Claude Code".

**Token discipline:** No design theory (craft is baked into Opus 4.7). No neutral
re-description of current visuals (capture sees them; describing them anchors + wastes
tokens) — current visuals appear only as keep/fix *verdicts*. No internal cross-links or
portfolio context in the pasteable block.

**`run-sheet.md`** — a separate, sequenced set of canvas follow-up prompts derived from the
site's *actual pages* (a redesign reuses the same pages with a new skin): paste onboarding →
validate (render palette swatch + one hero screen, judge the feel) → key screen → remaining
screens/components, ordered by the critique's top issues. Modeled on the MyPenFlow A–H
run-sheet, scaled to site complexity.

## 8. Category packs

Per-category best-practice files — the mechanism that feeds **category-specific taste**
(general craft is already in the model).

- **Location:** bundled locally at `plugins/reframe/categories/{category}.md`. No network,
  no versions, loaded by file read.
- **Proposed taxonomy (v1, ~10):** `local-service` (clinic/trades), `saas-marketing`,
  `ecommerce`, `portfolio-personal`, `editorial-blog`, `nonprofit`, `restaurant-hospitality`,
  `corporate-brochure`, `education-course`, `events`, plus **`generic.md`** fallback.
- **Each pack contains:** category redesign priorities; conversion patterns; trust signals;
  IA conventions; **opinionated design-system seed** (palette/type/motion stance);
  reference + anti-reference sites; strict NOs; "what to emphasize in the brief".
- **Detection:** from homepage + nav + content in Phase 7. **Low confidence → `generic.md`**
  (never hard-fail). **Multi-category sites → pick the dominant pack, note secondaries
  inline.** Never merge packs (produces contradictory guidance).

---

## 9. Coverage-first & inference model

Reconciles the user's **inference-first** preference (don't interview me) with correctness:

- **Inference-first is preserved as philosophy** — reframe infers; it does not interrogate.
- **Two mechanical gates** (not questions): the **render gate** (escalate to headless when
  fetched text is empty) and the **content-sufficiency gate** (greenfield mode when there's
  no real content). These guarantee coverage *before* inference.
- **Exactly one human question:** "same purpose or new?" — logically uninferable; one binary
  ask, not an interview.
- **Assumptions header** in every output makes wrong inferences visible and editable.
- **Confidence-gated category confirm is deferred** — v1 proceeds with best-guess category +
  an explicit noted assumption (the brief is editable output, not an irreversible action).

## 10. Error handling & graceful degradation

Unlike Beacon (which hard-stops rather than emit a partial API map), **reframe almost always
produces *something* and annotates confidence** — a brief from homepage + nav is still useful.

| Failure mode | Signal | Degrade rule |
|--------------|--------|--------------|
| SPA / empty server HTML | `[RENDER-ESCALATED]` | headless render before inferring |
| Placeholder / thin site | `[GREENFIELD-MODE]` | stop inferring; tell user it's not a redesign target |
| No sitemap | `[NO-SITEMAP]` | fall back to homepage nav crawl |
| Auth / paywall gated | `[COVERAGE-PARTIAL:gated]` | brief only public pages; name what's out of scope |
| Aggressive WAF/Cloudflare | `[WAF-BLOCKED]` | Firecrawl → Jina → browser-fetch; else partial + note |
| Huge / templated site | `[SAMPLED:n-templates]` | template-class sampling + "sampled, not exhaustive" |
| Single-page site | `[SINGLE-PAGE]` | section-level anchor map instead of page table |
| Multi-language / multi-brand | `[MULTI-LOCALE:canonical=x]` | brief one canonical locale/brand; name the rest |
| Chrome MCP unavailable | `[TOOL-UNAVAILABLE:chrome-mcp]` | text-only critique + explicit visual-gap note |

## 11. Packaging

Mirror Beacon's structure (skills auto-discovered by folder; no registration):

```
plugins/reframe/
├── .claude-plugin/plugin.json     ← name/version/description/author/keywords
├── skills/site-redesign/SKILL.md   ← lean (<15 KB); detail pushed to references/
├── commands/reframe-analyze.md     ← frontmatter name: reframe:analyze
├── categories/                     ← the ~10 + generic category packs
├── templates/                      ← INDEX / brief / run-sheet / inventory / ia-map / critique
├── references/                     ← crawl detail, tool-availability (copied+trimmed from Beacon)
├── README.md  CHANGELOG.md
```

No `schemas/`, `agents/`, `scripts/` for v1. No session-start hook (Beacon's rots; adds no
function). Do **not** replicate Beacon's phase-number bloat — 9 lean H2 sections.

**Build tooling.** Author the plugin with Anthropic's official tools, not by hand:
- **`plugin-dev`** (official, bundled in `anthropics/claude-code`; 7 plugin-authoring skills)
  — scaffolds `plugin.json`, command/skill layout, and validation. **Not currently installed
  here** → install/enable it before building, or fall back to mirroring Beacon's structure.
- **`skill-creator`** (official; available locally) — author and **eval** `skills/site-redesign/SKILL.md`
  (description-triggering accuracy + the process-invariant evals in §12).
Mirror Beacon only where these don't cover something.

## 12. Testing & eval

A prompt-driven skill produces *judgment*, not facts — so **assert on process invariants,
never on prose** (asserting on brief wording yields a flaky, useless suite). Reuse Beacon's
eval harness mechanically (`evals.json` / `grading.json`, old-skill vs with-skill runs).

**Process-invariant assertions:** category detected + logged; matching pack loaded
(`[PACK-LOADED:saas-marketing]`); render gate fired on a SPA fixture; greenfield gate fired
on a placeholder fixture; all output files written; the pivot question asked; a
`[WAF-BLOCKED]`/`[COVERAGE-PARTIAL]` signal appeared on a gated fixture; crawl stayed under
token budget.

**Minimal viable plan:** 2–3 fixtures of differing category + a live smoke run on
`trustyourphysio.com` (the SPA case).

---

## 13. v1 scope vs v2

**v1 (this spec):** the full 9-phase pipeline, visual track (render + screenshots),
evaluative content audit, IA/journey map, local category packs + generic fallback,
coverage-first gates + the one pivot question, `brief.md` **+ `run-sheet.md`**, KEEP register
+ redirect map + SEO-preservation, brand-asset keep list.

**v2 (deferred):** competitor/reference-site teardown; review-mining (GBP/Trustpilot) for
local services; owner-supplied analytics / Search Console input; post-Claude-Design
feedback/diff loop (evaluate generated design against brief goals → follow-up prompts);
interactive multi-question confirmation.

---

## 14. Risks

- **Inference quality on ambiguous sites** — mitigated by assumptions header + category
  fallback + the pivot question, not eliminated.
- **Chrome MCP availability** — required for SPA sites; degrades to text-only with a noted
  gap, but SPA briefs will be weaker without it.
- **Token cost of the visual track** — bounded by token-budgeted, template-sampled crawl.
- **Category-pack quality is the product's taste ceiling** — packs must be genuinely
  opinionated (seed values, references, anti-references), or briefs go bland.

## 17. Prior art & positioning (2026-06-21 web research)

A thorough sweep (Anthropic-official, community Claude skills/plugins, and commercial
tools) found **no tool occupying reframe's intersection**: *existing-URL in → strategic
purpose/audience/content/IA extraction → portable brief aimed at a downstream AI design
tool.* The niche is open. [confirmed across multiple direct negative queries]

**Adjacent camps (none are reframe):**
- **Anthropic official** — `frontend-design` (forward-only, no URL ingestion); `web-artifacts-builder`,
  `canvas-design`, `theme-factory`, `brand-guidelines` (visual generation). **Claude Design**
  imports a design system from *your* repo/files; it does **not** ingest an arbitrary live URL
  to extract strategy. A separate official `design` plugin does mockup critique, not live-URL analysis.
- **Closest community match** — `mistyhx/frontend-design-audit` (live URL → 15 usability
  heuristics + code fixes). Code-usability audit, not content/IA/strategy → brief. Also
  `SiteAudit MCP`, `claude-seo`, `sniff-qa` (SEO/perf/QA).
- **Commercial, three non-overlapping camps** — clone-and-redesign (Macaly, Shuffle, Playcode,
  Revamp, 10Web); UX/heuristic audits (UXLens, Heurilens, Talos); brief/IA generators (Relume —
  prompt-in/new-site, not existing-URL-in).

**Differentiation wedge:** strategy/semantic (not visual — Claude Design owns visual/tokens);
a *portable hand-off brief* (clone tools lock you into their builder); *existing-URL-in* (Relume
and brief generators start from a prompt). **Risk to respect:** the clone-redesign SaaS category
is large and well-funded — reframe must lean on the "brief for Claude Design specifically" wedge
and per-category analysis depth, not on a generic "redesign my site" pitch.

**Borrowed patterns** (folded into §5/§7/§8): Relume's IA-first input frame (reverse-engineered
from the site); `frontend-design-audit`'s severity-rated, principle-cited critique format; and
Claude Design's handoff-bundle shape with a **per-page intent triplet**.

**Build tooling discovered:** Anthropic ships an official **`plugin-dev`** plugin-authoring plugin
(in `anthropics/claude-code`) and the **`skill-creator`** skill — both should drive the build (§11).
