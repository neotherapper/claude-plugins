# SEO Plugin v0.1.0 — Design Document

**Date:** 2026-07-12
**Status:** Approved (Phase 2 docs PR)
**Scope:** v0.1.0 MVP — eight-phase site audit, scored reports, beacon/reframe integration, paidagogos curriculum

---

## Goal

Ship an SEO audit plugin that scores any site's SEO health, produces a structured report at `docs/sites/{slug}/seo/`, integrates with beacon and reframe, and teaches the underlying methodology via paidagogos. Zero API keys required for the core audit.

Out of scope for v0.1.0: live Core Web Vitals data, multi-page crawl, SERP/competitor analysis, GEO/AI-search tooling. Those are v0.2.0 → v1.0.0 in the roadmap.

---

## Non-goals

- **No dashboard** — reports are markdown files, not a UI. A future v1.0.0 dashboard would consume the per-site folders.
- **No historical scoring** — each run overwrites the prior 4 files. Trend tracking is v1.0.0 (`Historical Scoring`).
- **No real-time monitoring** — audits are on-demand via `/seo:audit`. Continuous CI is v1.0.0.
- **No multi-page crawl in v0.1.0** — single-page audit. v0.3.0 opens the crawl.

## Architecture

### Component map

```
        /seo:audit (command)
                 │
                 ▼
    skills/site-audit/SKILL.md   ◀── orchestrator (8 phases)
                 │
     ┌───────────┼─────────────┬──────────────┐
     ▼           ▼             ▼              ▼
   Phase 3    Phase 4       Phase 5        Phase 7
  meta.py   heading.py    schema.py  chrome-devtools MCP
                 │             │              +
                 ▼             ▼          nikai pagespeed
            scripts/*.py (Python, stdlib urllib)
                 │
                 ▼
            composite_scorer.py
                 │
                 ▼
            templates/*.md.template (token substitution)
                 │
                 ▼
            docs/sites/{slug}/seo/  (4 files)
```

### Three skills, not one

| Skill | Loads when | Surface |
|-------|-----------|---------|
| `site-audit` (orchestrator) | user runs `/seo:audit` | 8 phases, full report |
| `technical-audit` | user runs `/seo:technical` OR orchestrator Phase 6 invokes | crawlability, indexation, CWV |
| `on-page-audit` | user runs `/seo:on-page` OR orchestrator Phase 3–5 invokes | meta, headings, schema, content signals |

D-006 covers the rationale: context-budget isolation. A `/seo:technical` invocation should not pay the on-page loading cost, and a `/seo:audit` invocation should be able to load all three without paying for them three times.

### Why Python scripts (not MCP)

D-002 covers this. The plugin ships four executable scripts (`meta_audit.py`, `heading_audit.py`, `structured_data_validate.py`, `composite_scorer.py`) using the Python standard library only (`urllib.request`, `html.parser`, `json`, `argparse`). Reasons:

- No runtime to start — script invocation is `python3 script.py URL --json`, ~50 ms startup
- Inspectable — each script is < 200 lines and can be `cat`-read
- Combines with `argparse` so callers (`site-audit` SKILL.md) invoke them with deterministic arg lists
- No dependencies means no `pip install` for users — works on any Python 3.10+ environment

The MCP server pattern (`mcp-seo`) is rejected for v0.1.0 because it adds operational cost without solving a real problem at the 30-second audit size we ship.

## Scoring model

### Composite formula

```
total = technical·25 + on_page·25 + schema·20 + content·15 + performance·15
```

### Why these weights

- **Technical (25)** — highest single weight because technical SEO failures (broken sitemaps, canonical mistakes, no HTTPS) block indexing entirely. A site with meta tags and no sitemap gets no traffic.
- **On-Page (25)** — equal to technical because on-page signals are the developer-ownership zone: title, description, OG, Twitter, hreflang. These are the leverage points a developer can fix in an afternoon.
- **Structured Data (20)** — separate from on-page because the category is operationally distinct (validation logic differs) and high-value (rich results). Not higher because rich results are still <5% of clicks on average; mis-scoring it would overweight.
- **Content (15)** — captured here as heading-structure and image-alts; full content quality is out-of-scope and reputation/EEAT is not auditable programmatically.
- **Performance (15)** — kept small in v0.1.0 because live CWV data is gated on chrome-devtools MCP availability. When CWV is unavailable, this slot reads 0 and the audit still produces a graded report — users see what's missing.

### Per-rule thresholds (mirrored in `scoring-model.md` reference)

The thresholds in the scoring model reference table (50–60 char title, 150–160 char description, <2.5s LCP, <0.1 CLS, etc.) come from Google's published "Good" boundaries. We deliberately use Google's threshold, not a custom threshold — the audit must mean the same as Google Search Console's CWV alert, not some proprietary skew.

### Rating bands

| Range | Rating | Verbs (per the plan's intent) |
|-------|--------|--------------|
| 90–100 | Excellent | minor only |
| 70–89 | Good | address warnings |
| 50–69 | Needs Work | prioritise errors |
| 0–49 | Critical | overhaul |

These bands are exposed in `INDEX.md` so users can take a one-glance read of "do I need to spend the next hour on this, or ten?" without parsing five child scores.

### Deterministic top-5 selection

Top-5 issues in `INDEX.md` are **sorted**, not displayed in scan-order. The algorithm (documented in `skills/site-audit/references/top-issues.md`):

```
1. Pool all issues from every script (meta, heading, schema, tech, perf)
2. Sort by:
   - severity rank  (error=0, warning=1, info=2)
   - category-impact = max-points-in-category × (1 if failing-rule-is-in-that-category else 0)
   - then by rule-id alpha (stable tiebreaker)
3. Take first 5; if fewer, pad with "No further issues found"
4. Render as: [<severity>] <rule>: <message> (impact: <N> pts in <category>)
```

Deterministic selection makes the audit reproducible — two runs against the same site produce the same top-5.

## Integration

### Beacon → SEO (read)

```
beacon writes ─▶ docs/sites/{slug}/research/{INDEX,tech-stack,api-surfaces/*}.md
                       │
                       ▼
        seo/skills/site-audit/Phase 1 [RECON-REUSE] if tech-stack.md present
                       │
                       ▼
        skip redundant tech detection (D-008)
```

`beacon-integration.md` (reference file in `site-audit/references/`) documents:
- Where to read tech stack from
- When to skip which phases
- How to handle missing-but-expected data

### SEO → Reframe (write)

```
seo writes ─▶ docs/sites/{slug}/seo/{INDEX,seo-report,technical,on-page}.md
                       │
                       ▼
        reframe current-critique.md Step N reads and populates SEO section
                       │
                       ▼
        [REFRAME-REUSE] logged by reframe, not by SEO (SEO only writes)
```

`reframe-integration.md` documents the inverse: how a downstream consumer reads our output. Symmetry avoids surprises.

### Paidagogos ↔ SEO (teach + demonstrate)

```
paidagogos curriculum ─▶ 12 lessons ending with practical exercise
                              │
                              ▼
                    "/seo:audit your own site" (Lesson 12)
```

The curriculum lives in paidagogos/D-009. It references the plugin as a tool but imports nothing from `plugins/seo/`. Lesson 12 closes with: *"Run `/seo:audit` on your own site. Document the score, top 5 issues, and your fix plan."*

## Behavior under failure

The script-level bug fixes from the plan review (`meta_audit.py` OG-completeness branch, protocol-relative URLs, `heading_audit.py` whitespace, `composite_scorer.py` scoring-baseline, `structured_data_validate.py` deprecated-rule naming) all flow into the v0.1.0 source as-pasted in the plan's inline code blocks. Phase 0 of the plan is "bring the inline code up to spec"; no separate fix PR.

Tool failures are designed around:

| Failure | Behavior | Signal |
|---------|----------|--------|
| No MCP / no API keys | Audit proceeds with urllib + (optionally) nikai CLI | `[CWV-UNAVAILABLE]`, `[NO-API-KEYS]` |
| HTTP fetch timeout (>15s) | Abort after Phase 2 with clear error | `[FETCH-ERROR:timeout]` |
| nikai CLI missing | Skip enrichment, no error | `[CLI-UNAVAILABLE:{tool}]` |
| Python script fail | Mark category as `no-script` placeholder | `[SCRIPT-MISSING:{script}]` |
| Beacon research missing | Proceed with raw HTML tech inference | `[NO-RECON]` |
| Reframe current-critique missing | Don't try to update | (no signal — silent) |

This composition is documented in `free-tools.md` (Task 3) and the SKILL.md Phase tables.

## Open questions

1. **Performance (15 pts) — what does v0.1.0 score to?** With CWV unavailable, this category reads 0 by default. We accept that. v0.2.0 fixes it.
2. **Top-5 padding text** — when fewer than 5 issues, we render "No further issues found" per finding slot. That's verbose. Future: collapse to a single line.
3. **Token names in templates** — `{{H1_STATUS}}` etc. are computed by the orchestrator from rule metadata. If a rule gets a new severity rank, the status mapping needs updating. Tracking this isn't a v0.1.0 concern but is a future maintenance pain.
