**Version extraction:**
```bash
# No public version endpoint exists in Sylius 2.x — version must be inferred
# Attempt 1: check composer.lock via misconfigured exposure (rare)
curl -s {site}/composer.lock | python3 -c "import sys,json; d=json.load(sys.stdin); [print(p['version']) for p in d['packages'] if p['name']=='sylius/sylius']"

# Attempt 2: check Symfony debug toolbar (dev environments only)
curl -sI {site} | grep -i "X-Debug-Token"

# Attempt 3: check API entrypoint for API Platform version hint
curl -s {site}/api/v2/ -H "Accept: application/ld+json"

# Attempt 4: Wappalyzer via browser MCP — most reliable
```

## 2. Default API Surfaces

Sylius 2.x is built on **API Platform 3.x**. Default response format is **JSON-LD + Hydra** (`application/ld+json`). Collections return `hydra:member` arrays. Use `Accept: application/ld+json` for full Hydra responses; `Accept: application/json` for simplified JSON (some endpoints may have issues — see Gotchas).

Base URL: `{site}/api/v2/`

## 4. Auth Patterns

| Pattern | Location | Notes |
|---------|----------|-------|
| Customer JWT | `POST /api/v2/shop/customers/token` | Email + password; returns JWT + customer IRI |
| Admin JWT (2.x) | `POST /api/v2/admin-authentication-token` | Email + password; returns JWT for admin endpoints |
| Admin JWT (1.x fallback) | `POST /api/v2/admin/authentication-token` | Alternate path; present on some upgraded installs |
| Symfony session cookie | `PHPSESSID` | Browser-side; for storefront HTML pages |
| Bearer token header | `Authorization: Bearer <token>` | Used with all JWT-protected API endpoints |

**Customer JWT acquisition:**
```bash
curl -X POST {site}/api/v2/shop/customers/token \
  -H "Content-Type: application/json" \
  -d '{"email": "user@example.com", "password": "password"}'
# Response: {"token": "<jwt>", "customer": "/api/v2/shop/customers/{id}"}
```

**Admin JWT acquisition:**
```bash
# Sylius 2.x canonical endpoint (hyphenated path, NOT under /admin/):
curl -X POST {site}/api/v2/admin-authentication-token \
  -H "Content-Type: application/json" \
  -d '{"email": "api@example.com", "password": "sylius-api"}'
# Response: {"token": "<jwt>"}
# Default demo credentials: api@example.com / sylius-api
# Fallback for upgraded 1.x installs: /api/v2/admin/authentication-token
```

**Authenticated admin request:**
```bash
curl {site}/api/v2/admin/orders \
  -H "Authorization: Bearer <jwt>" \
  -H "Accept: application/ld+json"
```

## 6. Source Map Patterns

Sylius uses Symfony's Webpack Encore for asset compilation. Production deployments often strip source maps, but staging or misconfigured environments may expose them. Check:
- `{bundle}.js.map` — for all bundles listed in Section 5
- `/build/manifest.json` — Encore asset manifest; reveals all compiled asset paths
- `/build/entrypoints.json` — lists all entry points and their CSS/JS dependencies

If source maps are present, `sources[]` entries reveal original component file paths and may expose third-party plugin packages by name.

## 10. Gotchas

- **Default API format is JSON-LD + Hydra, not HAL.** Sylius 2.x uses API Platform 3.x which defaults to `application/ld+json`. Collections return `hydra:member` arrays with `@context`, `@id`, and `hydra:totalItems`. HAL (`application/hal+json`) and raw `application/json` are available via `Accept` header content negotiation, but some endpoints (notably shipping method normalization in Sylius 2.1) have bugs with `application/json` — always prefer `application/ld+json` for reliability.

- **Two separate JWT namespaces.** Shop customers authenticate at `/api/v2/shop/customers/token`; admin users authenticate at `/api/v2/admin-authentication-token` (Sylius 2.x canonical). A shop JWT will be refused on admin endpoints and vice versa. Do not mix them.

- **Admin authentication token path changed in 2.x.** In Sylius 1.x the path was `/api/v2/admin/authentication-token` (under the `/admin/` namespace). In Sylius 2.x the canonical path is `/api/v2/admin-authentication-token` (hyphenated at the v2 root). Stores upgraded from 1.x may respond on the old path. Probe both: if the 2.x path returns 404, try the 1.x variant. The task spec lists a third variant (`/api/v2/admin/administrators/token`) which appears in older community docs but is not confirmed in official 2.x documentation — treat as a last-resort fallback.

- **Locale-prefixed URLs are the default.** The storefront HTML uses `/{locale}/` URL prefixes (`/en_US/`, `/pl_PL/`). Omitting the prefix causes a redirect or 404 depending on configuration. The locale prefix format is `^[a-z]{2}(?:_[A-Z]{2})?$` (e.g., `en_US`, `en`, `pl_PL`). Some sites disable locale prefixes — detect by checking whether `/{locale}/` returns a page or a 404.

- **API endpoints do NOT use locale prefixes.** Unlike the storefront HTML, the REST API at `/api/v2/` is not locale-prefixed. Locale is specified via request headers (e.g., `Accept-Language`) or derived from the active channel config.

- **Sylius uses channels like Shopware uses sales channels.** A single Sylius installation can serve multiple independent storefronts (channels), each with its own domain, locale set, currency, and product catalog. The `sylius_channel` cookie identifies the active channel. When probing multi-channel sites, include the correct `Host` header or use the channel-specific domain.

- **Pagination cap.** API Platform enforces a maximum `itemsPerPage` (typically 100, sometimes lower). Requesting more than the cap silently returns the cap. Always read `hydra:totalItems` from the first page and calculate pages. Use `?page=2&itemsPerPage=100` to iterate.

- **ShopApiPlugin is deprecated in Sylius 2.x** but may still be active on stores that upgraded from 1.x without removing it. If `/shop-api/` returns 200, treat it as bonus surface — document endpoints but prefer `/api/v2/` equivalents.

- **No public version endpoint.** Sylius 2.x exposes no HTTP endpoint that reveals the exact version. Inference options: exposed `composer.lock` (misconfiguration), Symfony debug toolbar in dev environments, or Wappalyzer fingerprinting. Note the version as "2.x (inferred)" in your findings unless confirmed.

- **Admin path is configurable.** The default admin path is `/admin/` (login at `/admin/login`), but it is a configurable prefix in `config/routes/sylius_admin.yaml`. Some security-conscious deployments move it to an obscure path. If `/admin/login` returns 404, try common alternatives or check page source for admin links.

- **Sylius Plus vs Community Edition.** Sylius Plus (enterprise) adds endpoints not present in the open-source edition: `/api/v2/admin/returns/`, `/api/v2/admin/business-units/`, `/api/v2/admin/multi-source-inventory/`. The presence of these endpoints definitively identifies Sylius Plus.

- **Channel requirement on some admin endpoints.** Some admin API endpoints require a `SYLIUS-Store` or channel header in certain Sylius configurations. If a 400 or 422 is returned on admin endpoints despite a valid JWT, check whether a channel header is required.

- **CORS and OPTIONS preflights.** Sylius 2.x API Platform responses include CORS headers when configured. An `OPTIONS` preflight to `/api/v2/shop/products` may return different allowed headers than the actual `GET` — do not infer supported auth schemes from OPTIONS alone.

- **Sylius Fixtures in demo environments.** The Sylius demo (`v2.demo.sylius.com`) uses fixture data with predictable credentials (`sylius@example.com`/`sylius`, `api@example.com`/`sylius-api`). Production sites will have different credentials but may reuse default fixture product codes like `SYLIUS_MUG`, `SYLIUS_BOOK` — useful for probing product endpoints by code.
