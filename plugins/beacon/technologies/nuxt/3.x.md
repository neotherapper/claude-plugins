# Nuxt 3.x — Tech Pack

## 1. Fingerprinting Signals

| Signal | Type | Value | Confidence |
|--------|------|-------|------------|
| `/_nuxt/` path in HTML | HTML substring | `/_nuxt/` | High |
| `__NUXT_DATA__` inline JSON | HTML inline script | `__NUXT_DATA__` | High (Nuxt 3 payload) |
| `window.__NUXT__` object | JS global | `window.__NUXT__` | Medium (bridge object in Nuxt 3) |
| `x-powered-by: Nuxt` | Response header | `Nuxt` | High (if not disabled) |
| `nuxt-island` custom element | HTML element | `<nuxt-island` | High (Server Components, 3.1+) |
| Wappalyzer result | MCP | `"Nuxt.js"` | Definitive |

**Version extraction:**
```bash
# Step 1: extract buildId from window.__NUXT__ in the page HTML
SITE="https://example.com"
BUILD_ID=$(curl -s "$SITE" | grep -oP '"buildId"\s*:\s*"\K[^"]+')

# Step 2: fetch the build manifest using the real buildId
curl -s "$SITE/_nuxt/builds/meta/$BUILD_ID.json" | python3 -c "
import sys, json
data = json.load(sys.stdin)
print(data.get('nuxtVersion', 'not found'))
"
```

## 2. Default API Surfaces

| Endpoint | Method | Auth | Notes |
|----------|--------|------|-------|
| `/api/*` | GET/POST | Varies | Nuxt server routes (Nitro, defined in `server/api/`) |
| `/_nuxt/` | GET | None | Static assets directory — JS chunks, CSS |
| `/_nuxt/builds/latest.json` | GET | None | Build metadata — contains buildId and timestamp |
| `/_nuxt/builds/meta/{buildId}.json` | GET | None | Build manifest — modules, routes (sometimes protected) |
| `/_nuxt/manifest.json` | GET | None | Client-side routing manifest (may 404 if not public) |
| `/sitemap.xml` | GET | None | Present if `@nuxtjs/sitemap` is installed |
| `/sitemap_index.xml` | GET | None | Multi-sitemap index if `@nuxtjs/sitemap` is installed |
| `/robots.txt` | GET | None | Present if `@nuxtjs/robots` is installed |

## 3. Config / Constants Locations

| Location | How to access | Contains |
|----------|--------------|---------|
| `window.__NUXT__` | JS eval in browser / inline HTML script | Runtime config bridge, public config, page data |
| `window.__NUXT__.config.public` | JS eval | Explicitly public runtime config values |
| `/_nuxt/builds/latest.json` | Public GET | buildId, build timestamp |
| `/_nuxt/builds/meta/{buildId}.json` | Public GET (if not protected) | Routes, modules list, Nuxt version |
| `/_nuxt/manifest.json` | Public GET (if exposed) | All client-side route paths |
| `.nuxt/` directory | Server-side only | Not publicly accessible |
| `nuxt.config.ts` | Source only | Not publicly accessible |

## 4. Auth Patterns

| Pattern | Location | Notes |
|---------|----------|-------|
| `nuxt-auth` (sidebase) routes | `/api/auth/*` | Mirrors NextAuth.js; session at `/api/auth/session` |
| Nuxt Auth Utils routes | `/api/_auth/` | Alternative auth module by Atinux |
| Session cookie | `nuxt-session` or `__nuxt_session` | `nuxt-session` module |
| Bearer token (Nitro H3) | `Authorization` header | Common pattern in custom Nitro server routes |
| Misconfigured `useCookie()` | `window.__NUXT__` payload | Cookie values exposed if passed to state without filtering |

**Auth discovery notes:**
`nuxt-auth` (sidebase) is the most common auth layer and exposes `/api/auth/session`
and `/api/auth/providers` without authentication. Always probe these first.
Nuxt has no built-in auth — any auth mechanism is module-provided or custom Nitro routes.
Check `window.__NUXT__.config.public` for exposed auth provider names or OAuth client IDs.

## 5. JS Bundle Patterns

| Path | Content |
|------|---------|
| `/_nuxt/entry.{hash}.js` | App entry point — bootstraps Vue + router |
| `/_nuxt/{name}.{hash}.js` | Route-specific code-split chunks |
| `/_nuxt/{vendor}.{hash}.js` | Vendor bundle (Vue, Pinia, etc.) |

**Chunk URL extraction (two-step — never glob):**
```bash
# Step 1: extract all modulepreload hrefs from the HTML
SITE="https://example.com"
CHUNKS=$(curl -s "$SITE" | grep -oP '(?<=<link rel="modulepreload" href=")[^"]+')
echo "$CHUNKS"

# Step 2: fetch each chunk (example loop)
for CHUNK in $CHUNKS; do
  echo "=== $CHUNK ==="
  curl -s "$SITE$CHUNK" | head -5
done
```

Also check `/_nuxt/manifest.json` — it lists all route chunks with their hashed filenames.

## 6. Source Map Patterns

Source maps for Nuxt 3 production builds are disabled by default unless `sourcemap: true`
(or `sourcemap: { server: true, client: true }`) is set in `nuxt.config.ts`.

**Two-step probe (never glob):**
```bash
# Step 1: extract a real chunk filename from the HTML
SITE="https://example.com"
CHUNK=$(curl -s "$SITE" | grep -oP '(?<=<link rel="modulepreload" href=")/_nuxt/[^"]+\.js' | head -1)

# Step 2: probe the .map file for that specific chunk
curl -I "$SITE${CHUNK}.map"
```

If a `.map` file returns 200, recover original source paths from `sources` array inside the map JSON.
Check both entry chunk and route chunks — misconfigured CI pipelines sometimes ship maps for specific chunks only.

## 8. Known Public Data

| Endpoint | Data | Notes |
|----------|------|-------|
| `window.__NUXT__` (inline HTML) | buildId, public runtime config, initial page state | Read from raw HTML before any JS executes |
| `window.__NUXT__.state.$pinia` (inline HTML) | All Pinia store state | Often over-exposes: user session data, server-fetched API responses |
| `/_nuxt/builds/latest.json` | buildId, build timestamp | Always try; usually public |
| `/_nuxt/builds/meta/{buildId}.json` | Nuxt version, installed modules list, route manifest | Try after getting buildId; sometimes protected |
| `/_nuxt/manifest.json` | All client route paths with their chunk files | Try directly; may 404 if not published |
| `/api/_content/query?_params={}` | Nuxt Content CMS query — returns all content | v2 only; v1 used `/api/_content/` |
| `/api/auth/providers` | nuxt-auth provider list (GitHub, Google, etc.) | No auth required; always returns if nuxt-auth is installed |

## 9. Probe Checklist

Run these in order. Record result (200 / 403 / 404) for each.

- [ ] Read `window.__NUXT__` from page HTML: `curl -s $SITE | python3 -c "import sys, re; m = re.search(r'__NUXT__\s*=\s*(\{.+?\})\s*;', sys.stdin.read(), re.DOTALL); print(m.group(1)[:500] if m else 'not found')"` — get buildId, public config
- [ ] `GET /_nuxt/builds/latest.json` — build metadata; extract `buildId` for next step
- [ ] `GET /_nuxt/builds/meta/{buildId}.json` (use real buildId from step above) — full build manifest; reveals installed modules and Nuxt version
- [ ] `GET /_nuxt/manifest.json` — client route manifest; lists all route paths and their chunk filenames
- [ ] Check `<link rel="modulepreload">` tags in HTML — extract all chunk URLs for bundle analysis and source map probing
- [ ] `GET /api/auth/session` — nuxt-auth (sidebase) present? Returns session if logged in
- [ ] `GET /api/auth/providers` — nuxt-auth provider list; no auth required; confirms nuxt-auth install
- [ ] Check `window.__NUXT__.state.$pinia` in raw HTML — Pinia state exposure; may contain user data or server-fetched secrets
- [ ] `GET /api/_content/query?_params={}` — Nuxt Content v2 present? Returns CMS data if installed
- [ ] `GET /api/_auth/session` — Nuxt Auth Utils present? Returns authenticated user object
- [ ] `GET /sitemap.xml` — @nuxtjs/sitemap installed? Contains all public page paths
- [ ] `GET /robots.txt` — @nuxtjs/robots configuration; may reveal disallowed paths

## 10. Gotchas

- **`window.__NUXT__` in Nuxt 3 is NOT the Nuxt 2 object.** In Nuxt 3 the main page payload lives in `__NUXT_DATA__` (devalue-serialized), and `window.__NUXT__` is a smaller bridge object containing only config and build metadata. Parsing Nuxt 3 `__NUXT_DATA__` requires the `devalue` library — JSON.parse will fail on it.
- **Pinia state is routinely over-exposed.** `window.__NUXT__.state.$pinia` is serialized into the HTML for SSR hydration. Developers often pass full server-side API responses (including tokens, user objects, environment-leaked data) into Pinia stores without realising this ends up in raw HTML. Always read it.
- **`/_nuxt/builds/meta/{buildId}.json` access varies by deployment.** On some hosts (Vercel, Netlify) this file is public by default; on others it is intentionally blocked. If accessible, it is the fastest way to enumerate all installed Nuxt modules without reading source code.
- **`/api/` does NOT return a route listing.** Nitro returns 404 for unknown routes — it does not expose a route directory. Enumerate routes via `/_nuxt/manifest.json` or the build manifest, not by probing `/api/` directly.
- **Nitro deployment target affects behaviour.** Nuxt 3 apps can be deployed as Node.js server, Cloudflare Worker, Vercel Edge Function, or static. The `/api/` path is always consistent, but latency, response headers, and cold-start behaviour vary. A 500 on `/api/` may be a cold-start, not a real error.
- **`nuxt-content` v1 vs v2 query API differs.** The `/api/_content/query` endpoint is v2 only. Nuxt Content v1 (used with Nuxt 2) served queries at `/api/_content/`. Testing `/api/_content/query` on a v1 installation will always 404.
- **Source maps are build-time opt-in.** Unlike some frameworks, Nuxt 3 does not ship source maps in production by default. The absence of a `.map` file is expected, not evidence of intentional hiding — but always probe after extracting real chunk filenames.
