# Medusa.js 2.x — Tech Pack

## 1. Fingerprinting Signals

| Signal | Type | Value | Confidence |
|--------|------|-------|------------|
| `/health` returns `{"status":"ok"}` | HTTP response body | `{"status":"ok"}` | High (combined with other signals) |
| `/store/products` returns JSON array | HTTP response body | e-commerce JSON array | High (with e-commerce context) |
| `X-Medusa-*` response headers | HTTP header | prefix match | High |
| `medusa` in `package.json` or webpack chunks | JS bundle | string match | High |
| `@medusajs/` package refs in frontend JS chunks | JS bundle | `@medusajs/` import | Definitive |
| `publishable_api_key` in JS globals or env vars | JS global / env | key name match | Definitive |
| `NEXT_PUBLIC_MEDUSA_BACKEND_URL` in page source | HTML / JS | env var name | Definitive |
| Next.js `/_next/` paths with `__NEXT_DATA__` global | HTML / JS | global name | High (storefront only) |
| Dashboard at `/app/` (Medusa Admin) | URL path | `/app/` | High |
| Wappalyzer result | MCP | `"Medusa"` | Medium (if present) |

**Version extraction:**
```bash
# From npm package info embedded in frontend chunks:
curl -s {site}/_next/static/chunks/*.js | grep -oP '"@medusajs/medusa":"~?\K[^"]+'
# From response headers:
curl -sI {site}/health | grep -i "medusa\|x-powered-by"
```

## 2. Default API Surfaces

| Endpoint | Method | Auth | Notes |
|----------|--------|------|-------|
| `/health` | GET | None | Health check — returns `{"status":"ok"}`; may be disabled |
| `/store/products` | GET | Publishable key | Product list (paginated); no customer auth required |
| `/store/products/{id}` | GET | Publishable key | Single product with variants array |
| `/store/products?category_id[]={id}` | GET | Publishable key | Products filtered by category |
| `/store/products?limit=100&offset=0` | GET | Publishable key | Paginated full catalog (limit/offset, not cursor) |
| `/store/products?q={query}` | GET | Publishable key | Search (uses Meilisearch if configured) |
| `/store/collections` | GET | Publishable key | Product collections |
| `/store/categories` | GET | Publishable key | Category tree |
| `/store/regions` | GET | Publishable key | Available regions and currencies |
| `/store/carts` | POST | Publishable key | Create guest cart; returns cart ID |
| `/store/carts/{id}` | GET | Publishable key | Cart state |
| `/store/carts/{id}/line-items` | POST | Publishable key | Add item to cart |
| `/store/shipping-options?cart_id={id}` | GET | Publishable key | Available shipping options |
| `/store/payment-providers` | GET | Publishable key | Lists payment integrations (e.g. stripe) |
| `/store/customers` | POST | Publishable key | Create customer account |
| `/store/customers/me` | GET | Customer JWT | Current customer profile |
| `/auth/customer/emailpass` | POST | None | Customer JWT token (v2 auth route) |
| `/auth/user/emailpass` | POST | None | Admin/user JWT token (v2 auth route; separate from customer JWT) |
| `/admin/products` | GET | Admin JWT | Full product admin list |
| `/admin/orders` | GET | Admin JWT | Order management |

> **Note:** All `/store/` endpoints require the `x-publishable-api-key` header. Admin endpoints at `/admin/` require a separate admin JWT. Do not confuse the two.

## 3. Config / Constants Locations

| Location | What's there | How to access |
|----------|-------------|---------------|
| `NEXT_PUBLIC_MEDUSA_BACKEND_URL` in page source | API server URL (may differ from storefront domain) | HTML scrape / JS eval |
| `NEXT_PUBLIC_MEDUSA_PUBLISHABLE_KEY` in page source | Publishable API key | HTML scrape / JS eval |
| `__NEXT_DATA__` global | Next.js props including config and env | JS eval in browser |
| `publishableKey` runtime config in page source | Publishable API key (alternate embedding) | HTML scrape |
| `window.__MEDUSA_CONFIG__` | Runtime config (some deployments) | JS eval in browser |
| `medusa-config.js` / `medusa-config.ts` | Backend config: DB, modules, plugins | Source only |

**Extracting the publishable API key:**
```bash
# From Next.js embedded env vars:
curl -s {site} | grep -oP '"publishable_api_key"\s*:\s*"\K[^"]+'
# From __NEXT_DATA__ global:
curl -s {site} | grep -oP 'NEXT_PUBLIC_MEDUSA_PUBLISHABLE_KEY[^"]*"\K[^"]+'
# From runtime config in page source:
curl -s {site} | grep -oP 'publishableKey[^"]*"\K[^"]+'
# Reveal the actual API server URL:
curl -s {site} | grep -oP 'NEXT_PUBLIC_MEDUSA_BACKEND_URL[^"]*"\K[^"]+'
```

## 4. Auth Patterns

| Pattern | Header / Location | Notes |
|---------|------------------|-------|
| Publishable API key | `x-publishable-api-key: {key}` | Required for all `/store/` endpoints; not a secret — embedded in frontend |
| Customer JWT | `Authorization: Bearer {token}` | Obtained from `POST /auth/customer/emailpass`; required for `/store/customers/me` |
| Admin JWT | `Authorization: Bearer {token}` | Obtained from `POST /auth/user/emailpass`; entirely separate from customer JWT |
| Cookie session | Session cookie | Browser-only alternative to JWT for customer auth |

**Customer token acquisition:**
```bash
curl -X POST '{backend_url}/auth/customer/emailpass' \
  -H 'Content-Type: application/json' \
  -d '{"email": "customer@example.com", "password": "password"}'
```

**Admin/user token acquisition:**
```bash
curl -X POST '{backend_url}/auth/user/emailpass' \
  -H 'Content-Type: application/json' \
  -d '{"email": "admin@example.com", "password": "adminpassword"}'
```

| Path | Content |
|------|---------|
| `/_next/static/chunks/*.js` | Next.js storefront chunks (contain `@medusajs/` imports) |
| `/_next/static/chunks/pages/*.js` | Page-level bundles |
| `/__NEXT_DATA__` (inline `<script>`) | Server-side props, env vars, publishable key |
| Webpack chunk filenames containing `medusa` | Framework detection signal |
| `@medusajs/js-sdk` references in chunks | Medusa JS SDK usage |

Check each bundle URL + `.map` suffix for source maps.

## 6. Source Map Patterns

Medusa backend (Node.js) does NOT ship source maps in production.
Storefront frontends (Next.js Starter, Remix Starter) built with webpack/turbopack MAY have source maps.
Check: `{bundle-path}.js.map` — if 200, recover original TypeScript source paths and component structure.
The Next.js starter often compiles with source maps enabled by default in development; production deployments vary.

## 10. Gotchas

- **Backend and storefront are almost always on different domains.** `NEXT_PUBLIC_MEDUSA_BACKEND_URL` in page source reveals where the actual API server lives (e.g. `api.store.com` or `:9000`). Always extract it before probing API routes.
- **Publishable API key is required even for public store endpoints.** Omitting `x-publishable-api-key` returns 401/403 even on unauthenticated routes. The key is not a secret — it is embedded in the frontend.
- **Customer JWT and Admin JWT are entirely separate.** `POST /auth/customer/emailpass` gives a customer token; `POST /auth/user/emailpass` gives an admin token. They share the same `/auth/{actor}/{provider}` shape but are scoped to different actors.
- **Auth routes changed significantly in v2.** The v1 customer route was `POST /store/auth`; v2 uses `POST /auth/customer/emailpass`. The v1 admin route was `POST /admin/auth/token`; v2 uses `POST /auth/user/emailpass`. Do not assume v1 patterns work on v2 deployments.
- **Products with multiple variants return a single object.** The `variants` array is nested inside the product — there is no separate storefront variant endpoint.
- **Pagination is limit/offset, not cursor-based.** Use `?limit=100&offset=0`, `?limit=100&offset=100`, etc. to page through large catalogs.
- **Region selection affects pricing.** Always set a region when creating a cart if prices differ by market — use IDs from `GET /store/regions`.
- **Medusa Cloud may restrict admin routes.** Focus on the store API for reconnaissance; `/admin/` endpoints may return 404 or redirect on managed deployments.
- **`/health` may be disabled.** Some hardened deployments remove the health endpoint; its absence does not rule out Medusa.
- **Some deployments run admin on a separate subdomain.** If `/app/` returns 404 on the storefront domain, try the API server domain directly.
- **Meilisearch must be configured separately.** `GET /store/products?q=` falls back to basic filtering if Meilisearch is not installed — it will still return results, just unranked.
