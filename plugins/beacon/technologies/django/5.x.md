# Django 5.x Tech Pack

Django is a high-level Python web framework following the MVT (Model-View-Template) pattern. This tech pack covers fingerprinting, API surface discovery, and reconnaissance patterns for Django 5.x applications.

## 1. Fingerprinting Signals

| Signal | Type | Value | Confidence |
|--------|------|-------|------------|
| `csrfmiddlewaretoken` hidden input in HTML forms | HTML | `<input type="hidden" name="csrfmiddlewaretoken" value="...">` | High — Django CSRF middleware |
| `sessionid` cookie | HTTP Cookie | `sessionid=<value>; HttpOnly` | High — Django default session cookie name |
| `django.min.js` or `django.js` in page source | HTML script tag | `/static/admin/js/django.min.js` | Definitive — Django admin active |
| `Server` response header | HTTP Header | `WSGIServer/2.0` or `gunicorn/...` | Medium — common Django deployment servers |
| `__debug__` toolbar in HTML | HTML comment / DOM | `djdt` in HTML or `/__debug__/` route | High — Django Debug Toolbar active (dev only) |
| `X-Frame-Options: SAMEORIGIN` header | HTTP Header | `X-Frame-Options: SAMEORIGIN` | Medium — Django default security header |
| Django version comment in `/admin/` login page | HTML comment | `<!--Django v5.x-->` or error page stack trace | High when DEBUG=True |
| Wappalyzer detection | Browser extension | `"Django"` result | Medium — technology fingerprint |

**Version extraction (bash):**

```bash
# Check admin login page for version hints (works if DEBUG=True or version comment present)
curl -s https://target.example.com/admin/login/ | grep -i 'django'

# Check error page for version (only works if DEBUG=True — trigger a 404)
curl -s https://target.example.com/nonexistent-path-that-does-not-exist-abc123/ | grep -i 'django'
```

## 3. Config / Constants Locations

| Location | How to access | Contains |
|----------|---------------|----------|
| `settings.py` (server-side) | Not accessible remotely | DATABASE_URL, SECRET_KEY, INSTALLED_APPS, MIDDLEWARE, STATIC_URL, DEBUG |
| `csrfmiddlewaretoken` in HTML forms | Parse any page HTML | Per-request CSRF token for POST operations |
| `csrftoken` cookie | HTTP response cookies | CSRF cookie value; pass as `X-CSRFToken` header for AJAX |
| `STATIC_URL` setting | Inferred from static asset URLs (typically `/static/`) | Static files base URL |
| Django Debug Toolbar panels | `/__debug__/` routes (dev only) | DB queries, request info, settings, cache stats |
| `DJANGO_SETTINGS_MODULE` env var | Not accessible remotely | Points to active settings module |
| `/api/schema/` | HTTP GET (if drf-spectacular is public) | Full OpenAPI schema including all routes, parameters, models |

## 4. Auth Patterns

| Pattern | Location | Notes |
|---------|----------|-------|
| Session authentication | `sessionid` cookie | Default Django auth; HttpOnly + Secure in production |
| CSRF token | `csrfmiddlewaretoken` form field or `csrftoken` cookie | Required for all non-GET requests; send as `X-CSRFToken` header for AJAX/API calls |
| DRF Token authentication | `Authorization: Token <token>` header | Enabled when `rest_framework.authtoken` is in INSTALLED_APPS |
| DRF JWT (SimpleJWT) | `Authorization: Bearer <jwt>` header | Enabled when `djangorestframework-simplejwt` is installed |
| django-allauth | `/accounts/login/`, `/accounts/signup/` routes | Social and email/password auth |
| Django admin session | `/admin/login/` POST with `username`, `password`, `csrfmiddlewaretoken` | Separate admin session from regular user session |

**CSRF token acquisition for write operations:**

```bash
# Step 1: fetch login page to get CSRF token and session cookie
CSRF=$(curl -sc /tmp/dj_cookies.txt https://target.example.com/admin/login/ \
  | grep csrfmiddlewaretoken | grep -oP 'value="\K[^"]+')

# Step 2: POST login using both cookies and CSRF token header
curl -sb /tmp/dj_cookies.txt -c /tmp/dj_cookies.txt \
  -H "X-CSRFToken: $CSRF" \
  -d "username=admin&password=secret&csrfmiddlewaretoken=$CSRF&next=/admin/" \
  https://target.example.com/admin/login/
```

## 5. JS Bundle Patterns

| Path | Content |
|------|---------|
| `/static/admin/js/django.min.js` | Django admin core JS — presence confirms admin is active |
| `/static/admin/js/nav_sidebar.js` | Django admin sidebar JS (Django 3.1+) |
| `/static/rest_framework/js/default.js` | DRF browsable API JS — confirms DRF installed |
| `/static/{app}/js/{bundle}.js` | App-specific static JS files |
| `/static/manifest.json` | Webpack/Vite asset manifest — if a front-end build tool is used |
| `/static/webpack-stats.json` | Webpack bundle stats — if webpack-bundle-tracker is used |

Django does not bundle JavaScript natively. Static files are served as-is via `STATICFILES_DIRS` / `STATIC_ROOT`. In production, whitenoise or a CDN (S3, CloudFront) typically serves these files. If a separate front-end build tool (webpack, Vite) is configured, a manifest file maps logical names to hashed filenames.

## 6. Source Map Patterns

Django does not generate source maps natively. Source maps are only present if a front-end build tool (webpack, Vite, esbuild) is explicitly configured to emit them.

**Where to look:**

```bash
# Check for source maps alongside known JS files
curl -I https://target.example.com/static/admin/js/django.min.js.map

# If a manifest exists, resolve the hashed filename first, then check for map
curl -s https://target.example.com/static/manifest.json

# Example two-step: extract hashed filename from manifest, then fetch the map
# Step 1 — get manifest
MANIFEST=$(curl -s https://target.example.com/static/manifest.json)
# Step 2 — parse out filename for a known logical key (use jq or grep)
HASHED=$(echo "$MANIFEST" | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('main.js',''))")
# Step 3 — probe for its source map
curl -I "https://target.example.com/static/${HASHED}.map"
```

Static files fingerprinted by whitenoise follow the pattern `/static/{hash12}/{filename}.js`. Do not use glob patterns in curl — always resolve the exact filename first via the manifest or by parsing HTML source, then fetch the resolved path directly.

## 7. Common Plugins & Extensions

| Package | API it adds | Detection signal |
|---------|-------------|-----------------|
| Django REST Framework (DRF) | `/api/` routes, browsable API HTML, schema endpoints | `/api/` returns 200; `Accept: text/html` returns DRF browsable API page; `/static/rest_framework/` assets present |
| drf-spectacular | `/api/schema/`, `/api/schema/swagger-ui/`, `/api/schema/redoc/` | GET `/api/schema/` returns OpenAPI YAML/JSON |
| drf-yasg | `/swagger/`, `/redoc/`, `/swagger.json`, `/swagger.yaml` | GET `/swagger.json` returns OpenAPI spec |
| django-allauth | `/accounts/login/`, `/accounts/signup/`, `/accounts/` routes | GET `/accounts/login/` returns allauth login form |
| celery | No HTTP surface (background worker) | Detect indirectly via `/admin/` task result models or `django-celery-results` admin pages |
| django-storages + S3 | Static/media URLs redirect to S3 or CDN | Static file URLs contain `s3.amazonaws.com` or CloudFront domain |
| wagtail CMS | `/cms/` or `/wagtail/` admin, `/api/v2/` Wagtail content API | GET `/api/v2/pages/` returns Wagtail page list |
| django-debug-toolbar | `/__debug__/` routes, toolbar injected into HTML | GET `/__debug__/render_panel/` present; toolbar HTML in page source |
| django-cors-headers | `Access-Control-Allow-Origin` response header | CORS headers present on API responses |
| channels (WebSocket) | WebSocket endpoint (ws:// or wss://) | Upgrade: websocket header supported on certain paths |

|----------|------|-------|

| `/swagger.json` | Full OpenAPI spec in JSON | Only public if drf-yasg is configured without auth requirement |

## 9. Probe Checklist

- [ ] `GET /admin/` — Is the admin panel present? (302 to login = present at default URL; 200 = logged in already; 404 = custom URL or disabled — check `/robots.txt` and HTML source for hints)
- [ ] `GET /admin/login/` — Retrieve the admin login page; inspect HTML for Django version comments and `csrfmiddlewaretoken` field
- [ ] `GET /api/` — Is Django REST Framework installed? (200 with DRF browsable API HTML or JSON root = DRF active; 404 = not at this path, try `/api/v1/`)
- [ ] `GET /api/schema/` — Is drf-spectacular present? (200 with OpenAPI YAML = full API schema available publicly)
- [ ] `GET /api/schema/swagger-ui/` — Is the Swagger UI publicly accessible? (200 = interactive API explorer available)
- [ ] `GET /swagger.json` — Is drf-yasg present? (200 with JSON OpenAPI spec = drf-yasg installed and schema is public)
- [ ] `GET /swagger/` — Alternate drf-yasg Swagger UI path
- [ ] `GET /__debug__/` — Is Django Debug Toolbar reachable? (should return 404 in production; 200 or 404 with djdt content = debug mode active, information leak present)
- [ ] `GET /accounts/login/` — Is django-allauth installed? (200 with allauth login form = allauth active)
- [ ] `GET /accounts/signup/` — Is user self-registration enabled via allauth? (200 = registration open)
- [ ] `GET /static/rest_framework/js/default.js` — Confirms DRF static assets are present, independently verifying DRF installation
- [ ] `GET /sitemap.xml` — Is `django.contrib.sitemaps` active? (200 with XML = sitemap present, reveals public URL structure)
- [ ] `GET /robots.txt` — Check for custom admin URL or disallowed paths that reveal site structure
- [ ] `GET /api/v2/pages/` — Is Wagtail CMS installed? (200 with JSON page list = Wagtail content API active)

## 10. Gotchas

- **No standard API URL prefix.** Django has no built-in convention for API paths. `/api/` is community convention but many projects use `/v1/`, `/rest/`, `/service/`, or entirely custom paths defined in `urls.py`. A 404 at `/api/` does not mean DRF is absent — probe multiple prefixes and check `/robots.txt` and the OpenAPI schema for the actual path structure.

- **CSRF is required for write operations even with token auth.** When DRF uses `SessionAuthentication` (the default when a Django session cookie is present), every POST, PUT, PATCH, and DELETE request also requires a valid `X-CSRFToken` header. Always acquire the CSRF token from the `csrftoken` cookie or a form's `csrfmiddlewaretoken` field before any write operation — do not assume token authentication bypasses CSRF.

- **The admin URL is frequently moved.** Relocating the admin to a non-standard URL (e.g., `/dashboard/`, `/control/`, `/manage/`) is a common baseline security practice. A 404 at `/admin/` does not mean admin is disabled — check `/robots.txt` for a `Disallow:` entry pointing to the real admin path, and inspect HTML source of the home page for links.

- **DRF browsable API returns HTML by default.** The same DRF endpoint returns HTML when the request includes `Accept: text/html` and JSON when it includes `Accept: application/json`. Beacon probes must always send `Accept: application/json` (or `Accept: application/json, */*`) to get machine-readable API responses and avoid parsing an HTML page as a JSON object.

- **DEBUG=True is a major information source in dev/staging.** When `DEBUG=True`, Django error pages include the full stack trace, Django version, installed apps list, and all settings values. Intentionally triggering a 404 or 500 (e.g., by requesting a path with a malformed parameter) on a debug-mode site extracts version and configuration data that is otherwise server-side only.

- **Static files in production are often not served by Django.** Whitenoise, Amazon S3, or a CDN (CloudFront, Cloudflare) typically serves static files in production. The `/static/` URL base may redirect to an S3 bucket or CDN origin — follow `Location` headers on redirects to identify the true static asset host. When whitenoise is used with file fingerprinting, static file paths include a 12-character hash (`/static/abc123def456/app.js`); resolve the exact path via the asset manifest before fetching rather than guessing the hash.
