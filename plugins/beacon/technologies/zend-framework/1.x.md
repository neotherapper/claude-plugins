# Zend Framework 1.x — Tech Pack

> **EOL notice:** ZF1 reached end-of-life in September 2016 (last release: 1.12.20, 2016-06-20).
> Sites still running ZF1 are almost certainly on legacy infrastructure and may have
> unpatched vulnerabilities. Treat every finding here as higher-risk than equivalent
> modern-framework findings.

## 1. Fingerprinting Signals

| Signal | Type | Value | Confidence |
|--------|------|-------|------------|
| URL pattern `/controller/action` or `/module/controller/action` | URL structure | MVC triple | High |
| `index.php/controller/action` in URLs | URL path | Fallback route (no mod_rewrite) | Definitive |
| `X-Powered-By: PHP/5.x` header | HTTP header | PHP 5.2–5.6 common with ZF1 | Low (not ZF-specific) |
| Zend exception page in HTML | Error page | `Zend_Controller_Exception` or `Zend_Exception` in body | Definitive |
| `/application/` or `/library/Zend/` path leak | HTML/JS asset path | Source path in error stack | Definitive |
| `PHPSESSID` cookie | Cookie | Standard PHP session; ZF1 uses it natively | Low (not ZF-specific) |
| `ZDEDebuggerPresent` cookie | Cookie | Zend Debugger (Zend Studio / Zend Server) — co-located with ZF1 | High |
| `__hash__` hidden form field | HTML input | `Zend_Form_Element_Hash` default name — closest ZF1 has to CSRF | High |
| `X-Powered-By: Zend Server` or `Zend Guard` | HTTP header | Zend Server / Zend Guard deployment (common ZF1 host) | High |
| `application/configs/application.ini` accessible | File probe | ZF1 config file | Definitive |
| Wappalyzer result | MCP | `"Zend Framework"` | Definitive |
| `<meta name="generator" content="Zend Framework"` | HTML meta | Generator meta (rare) | Definitive |

**Version extraction:**
```bash
# ZF1 version is rarely exposed in HTTP headers; check error pages or exposed files
# Try the library version file (if document root misconfigured)
curl -s {site}/library/Zend/Version.php 2>/dev/null | grep -o "VERSION = '[^']*'"
# Or check for exposed composer.json
curl -s {site}/composer.json 2>/dev/null | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('require',{}).get('zendframework/zendframework1',''))" 2>/dev/null
# Force an error page to reveal version in stack trace
curl -s "{site}/nonexistent-module-xyz/action" | grep -i "zend"
```

## 2. Default API Surfaces

ZF1 has no built-in REST API. All HTTP surface is MVC controller-action routes.
The default router maps: `/{controller}/{action}/{param}/{value}...`
With modules enabled: `/{module}/{controller}/{action}/{param}/{value}...`

| Endpoint | Method | Auth | Notes |
|----------|--------|------|-------|
| `/` or `/index/index` | GET | None | Default home — IndexController::indexAction() |
| `/index/index` | GET | None | Explicit default action |
| `/{controller}/index` | GET | None | Default action for any controller |
| `/auth/login` | GET/POST | None | Typical login controller (convention) |
| `/auth/logout` | GET | Session | Typical logout action |
| `/admin/` or `/admin/index` | GET | Session/ACL | Admin module root (if admin module exists) |
| `/api/` | GET/POST | Varies | Custom API controller (common convention) |
| `/rest/` | GET/POST/PUT/DELETE | Varies | Zend_Rest_Server routes (if configured) |
| `/xmlrpc/` | POST | Varies | Zend_XmlRpc_Server routes (if configured) |
| `/error/error` | GET | None | Default error controller — triggered on exceptions |
| `index.php/{controller}/{action}` | GET | None | Fallback URL form when mod_rewrite is off |

**Param-value URL pattern:**
```bash
# ZF1 appends parameters as pairs: /controller/action/key1/value1/key2/value2
# Example: /products/list/category/electronics/page/2
curl -s "{site}/products/list/category/test/page/1"
```

## 3. Config / Constants Locations

| Location | What's there | How to access |
|----------|-------------|---------------|
| `application/configs/application.ini` | DB credentials, resource config, environment settings | File probe (often exposed — critical finding) |
| `application/configs/application.xml` | Alternative config format | File probe |
| `application/Bootstrap.php` | Application bootstrapper, resource registration | Source only |
| `.htaccess` (document root) | `RewriteRule` to `public/index.php`; sometimes exposes directory structure | File probe |
| `public/.htaccess` | Same as above in public-root setups | File probe |
| Error page body | Full stack trace with file paths, class names, and config values in dev mode | Trigger a 404/500 |
| `window.zf` or inline JS | Config values passed from PHP to JS via `Zend_View` | HTML scrape |

**Probe for exposed config (critical):**
```bash
# These paths are commonly left world-readable in misconfigured ZF1 deployments
curl -s {site}/application/configs/application.ini
curl -s {site}/../application/configs/application.ini
curl -s {site}/config/application.ini
# Check if /application/ directory is browsable
curl -s -I {site}/application/
```

## 4. Auth Patterns

| Pattern | Location | Notes |
|---------|----------|-------|
| `Zend_Auth` session | `PHPSESSID` + `Zend_Auth` session namespace | Login via POST to auth controller; session persists identity |
| HTTP Basic / Digest | `Authorization` header | `Zend_Auth_Adapter_Http` — rare outside APIs |
| Database adapter | POST form to auth controller | `Zend_Auth_Adapter_DbTable` — most common; checks username/password against DB table |
| LDAP adapter | POST form or auto-login | `Zend_Auth_Adapter_Ldap` — enterprise deployments |
| ACL roles | Session identity matched against `Zend_Acl` | Roles/resources/privileges in Bootstrap or plugin |

**Typical login flow:**
```bash
# Step 1: GET login form — capture CSRF token if present (ZF1 has no built-in CSRF; plugin may add it)
curl -c cookies.txt -s {site}/auth/login > login.html
# Look for hidden input field with token name
grep -i 'hidden.*token\|csrf\|_token' login.html

# Step 2: POST credentials
curl -b cookies.txt -c cookies.txt -X POST {site}/auth/login \
  -d "username=admin&password=admin"
# Successful login: 302 redirect to dashboard; failed: back to login with error message

# Step 3: Verify session with authenticated route
curl -b cookies.txt -s {site}/admin/index
```

## 5. JS Bundle Patterns

ZF1 predates modern bundlers; JS is typically hand-crafted or jQuery-based.

| Path | Content |
|------|---------|
| `/public/js/` | Application JS (flat files, no bundles) |
| `/public/js/app.js` or `/public/js/main.js` | Main application script |
| `/js/` (at document root) | Older deployments without `/public/` separation |
| `/public/js/jquery*.js` | jQuery (1.x–1.12 common with ZF1) |
| `window.BASE_URL` or `window.baseUrl` | Base URL injected from PHP (grep HTML for this) |

No source maps expected — ZF1 era predates source map generation.

## 6. Source Map Patterns

Source maps are not present in ZF1 applications. Skip Phase 7 source map checks.

The equivalent intel source in ZF1 is **exposed error pages** — in development or
staging mode, ZF1 outputs full PHP stack traces including file paths, class names,
and sometimes config values. Trigger deliberately:

```bash
# Request a guaranteed 404 to trigger Zend error controller
curl -s "{site}/zf1-recon-probe-xyz/nonexistent" | grep -i "zend\|application\|stack\|exception"
```

## 7. Common Plugins & Extensions

| Package | API it adds | Detection signal |
|---------|------------|-----------------|
| `Zend_Rest_Server` | RESTful controller at `/rest/` | Route config in `application.ini` or Bootstrap |
| `Zend_XmlRpc_Server` | XML-RPC at `/xmlrpc/` | `POST /xmlrpc/` with `system.listMethods` call |
| `Zend_Json_Server` | JSON-RPC at `/jsonrpc/` | Content-Type: application/json-rpc |
| `Zend_Feed` | `/feed/`, `/rss/`, `/atom/` routes | HTML `<link rel="alternate">` |
| `ZFDebug` / `Zend_Debug` | Debug toolbar injected in HTML | `id="ZFDebug"` div in page footer; `<!-- ZFDebug -->` comment (secondary) |
| `Doctrine 1.x` | ORM (no HTTP surface) | `/library/Doctrine/` path in error traces |
| `Zend_Search_Lucene` | Search controller (usually `/search/`) | Convention-based |
| `Zend_Captcha` | Form field on POST endpoints | `<img src="/captcha/...">` in HTML |
| `Zend_Gdata` | Google data API client (no HTTP surface) | Import path in errors |

## 9. Probe Checklist

- [ ] `GET /` — confirm base response; note redirect pattern
- [ ] `GET /zf1-probe-xyz/action` — trigger error page; look for ZF exception classes in response
- [ ] `GET /application/configs/application.ini` — check for exposed config (critical if 200)
- [ ] `GET /.htaccess` — check for exposed Apache config
- [ ] `GET /index.php` — confirm PHP entry point; note if `index.php/` prefix required (mod_rewrite off)
- [ ] `GET /index/index` — explicit default action
- [ ] `GET /auth/login` — locate login form; note hidden fields and POST target
- [ ] `GET /auth/logout` — test session destruction (expect redirect)
- [ ] `GET /admin/` or `GET /admin/index` — admin module (expect 302 to login if ACL-protected)
- [ ] `GET /api/` — check for custom REST controller
- [ ] `POST /xmlrpc/` with `<methodCall><methodName>system.listMethods</methodName></methodCall>` — XML-RPC surface
- [ ] `GET /feed/` and `GET /rss/` — RSS/Atom feeds
- [ ] `GET /error/error` — direct access to error controller (sometimes accessible)
- [ ] Check response headers for `X-Powered-By` PHP version
- [ ] Grep HTML source for `BASE_URL`, `baseUrl`, module/controller names in JS
- [ ] Check for `ZFDebug` toolbar in HTML response (dev environments)

## 10. Gotchas

- **No mod_rewrite = URL prefix required.** When `mod_rewrite` is disabled or misconfigured,
  all ZF1 URLs require `index.php/` prefix: `{site}/index.php/controller/action`. Always
  test both forms if the canonical form returns 404.

- **Module vs non-module setups.** Small apps use the default module only (`/controller/action`).
  Larger apps use named modules (`/module/controller/action`). The presence of `/admin/` or
  `/api/` as a three-segment path is a module hint, not a subdirectory.

- **`application.ini` exposure is endemic.** Many ZF1 deployments put the application root
  above the web root, but misconfigured Apache/Nginx can expose it. Always probe — it contains
  database credentials, SMTP passwords, and API keys.

- **Error pages reveal the full directory layout.** ZF1 dev-mode error pages print the PHP
  stack trace including absolute file paths (e.g., `/var/www/html/application/controllers/`).
  This maps the entire application structure from a single bad request.

- **No built-in CSRF protection.** ZF1 has no framework-level CSRF token. Some apps use
  `Zend_Form` with hash elements, but many POST endpoints accept cross-site requests.
  Check for hidden `hash` or `token` form fields; if absent, CSRF is likely unprotected.

- **Session fixation risk.** `Zend_Session` uses PHP's native session handler. Older
  deployments may not regenerate the session ID after login — test by comparing `PHPSESSID`
  before and after auth.

- **PHP 5.x-only (upstream).** Upstream ZF1 (1.12.20) is incompatible with PHP 8.x. However,
  the `shardj/zf1-future` community fork maintains PHP 7.x and 8.x compatibility and has real
  production deployments — a PHP 7/8 server does not conclusively rule out ZF1. If the server
  exposes PHP 5.x via `X-Powered-By`, treat the entire host as unpatched legacy infrastructure.

- **`Zend_Config_Ini` strips comments.** The `.ini` parser is strict — if you get the file,
  parse it as standard INI. Section headers like `[production : staging]` use ZF's inheritance
  syntax (read: `production` extends `staging`).

- **XML-RPC introspection.** If `/xmlrpc/` returns a valid XML-RPC fault for an unknown method,
  `system.listMethods` will return all available methods — this is the closest ZF1 equivalent
  to an OpenAPI spec.
