# PrestaShop 8.x — Tech Pack

PrestaShop is a major open-source PHP e-commerce platform dominant in France, Spain, Greece,
and broader Southern/Eastern Europe. PS 8.x runs on Symfony 4+. The Skroutz Marketplace
(Greece's #1 e-commerce platform, 80%+ market share) requires PrestaShop stores to expose
an XML product feed — making feed module enumeration especially valuable for Greek-market
`.gr` domains. Many stores on `.fr`, `.es`, `.gr`, `.pt`, `.pl` domains run PrestaShop.

## 1. Fingerprinting Signals

| Signal | Type | Value | Confidence |
|--------|------|-------|------------|
| `PrestaShop-{hash}` cookie | Cookie | Name prefix match | Definitive |
| `id_lang` cookie present | Cookie | Name match | High |
| `id_currency` cookie present | Cookie | Name match | High |
| `id_shop` cookie present | Cookie | Name match | High (multistore) |
| Generator meta tag | HTML meta | `<meta name="generator" content="PrestaShop">` | Definitive |
| `window.prestashop` JS global | JS global | Object present in inline scripts | Definitive (Classic theme) |
| `window.__PS_BASE_URI__` JS variable | JS global | Variable present in inline scripts | Definitive |
| `var prestashop = {...}` inline JS | HTML substring | Inline JS object in page source | Definitive |
| `/modules/` in HTML src attributes | HTML substring | Script or link src contains `/modules/` | High |
| `/themes/{theme-name}/assets/` in HTML | HTML substring | Script or link src contains `/themes/` | High |
| `id_product` or `id_category` in URL query | URL pattern | `?id_product=N` or `?id_category=N` | High |
| `/index.php?controller=` URL pattern | URL pattern | Controller-style URL present | High |
| `/{category}/{product}.html` URL pattern | URL pattern | Friendly URL with `.html` suffix | Medium |
| Wappalyzer result | MCP | `"PrestaShop"` | Definitive |

**Version extraction:**
```bash
# Generator meta tag (most reliable if present):
curl -s {site} | grep -oP 'content="PrestaShop \K[\d.]+'

# psws-version header from Web Service API probe (most reliable — works even without auth):
curl -sI {site}/api/ | grep -i "psws-version"
# → psws-version: 8.1.7  (401 = WebService enabled; 404 = disabled; any response = PrestaShop confirmed)

# From config file (source access only):
# /config/settings.inc.php contains define('_PS_VERSION_', 'x.y.z')

# Admin panel title (if /admin{hash}/ accessible):
curl -s {site}/admin{hash}/index.php | grep -oP 'PrestaShop \K[\d.]+'

# New Admin API docs header (PS 8.1+):
curl -sI {site}/admin-api/docs.html | grep -i "prestashop"
```

**Web Service API detection via `/api/`:**
Always probe `GET {site}/api/` and read both the response code AND the `psws-version` header:
- `401` + `psws-version: {ver}` → WebService enabled, version confirmed; log `[PRESTASHOP:wsv:{ver}:ws-enabled=true]`
- `404` → WebService disabled (normal for most production stores); log `[PRESTASHOP:ws-enabled=false]`
- Either response = PrestaShop confirmed. This is a stronger fingerprint than cookie or meta tag.

**Note on Hummingbird theme (PS 8.1+):** The new default theme removes jQuery-dependent
JS globals. `window.prestashop` may not be present in pages served by Hummingbird — use
cookie-based fingerprinting (`PrestaShop-{hash}`) as the primary signal instead. Data
previously in JS globals is now stored in `localStorage` (browser-side only, not accessible
via curl).

| Location | What's there | How to access |
|----------|-------------|---------------|

| Product page `<script type="application/ld+json">` | Name, price, currency, availability, SKU, image URL | HTML scrape — no auth required |
| `/admin-api/docs.html` | Full Swagger spec of Admin API endpoints | HTTP GET — may be accessible without auth |

## 4. Auth Patterns

| Pattern | Location | Notes |
|---------|----------|-------|
| Web Services API Key — Basic auth | `Authorization: Basic base64(KEY:)` | HTTP Basic; API Key as username, empty password; generated in Admin > Advanced Parameters > Webservice |
| Web Services API Key — query param | `?ws_key={KEY}` | Functional alternative to Basic auth; appears in server logs — avoid for stealth |
| Customer session cookie | `PrestaShop-{hash}` | Front-office session; set on first visit; required for cart state |
| Admin session cookie | `psAdmin` | Back-office session; short-lived |
| CSRF token — forms | `static_token` hidden field | Present in front-office forms; required for POST actions |
| CSRF token — admin URLs | `token` GET parameter | Admin panel URLs include `&token={hash}` |
| JWT token (Admin API) | `Authorization: Bearer {token}` | PS 8.1+ Admin API; acquire via `POST /api/v2/token` with client_id/client_secret |
| `PHPSESSID` cookie | Cookie | Standard PHP session ID; always present |

**Web Services API Key usage:**
```bash
# List all accessible resources with permissions:
curl -u "{API_KEY}:" "{site}/api/?output_format=JSON"

# Fetch product list — FULL fields (not just ID+link):
curl -u "{API_KEY}:" "{site}/api/products?output_format=JSON&display=full&limit=10"

# Discover schema without fetching data (fastest field enumeration):
curl -u "{API_KEY}:" "{site}/api/products?schema=synopsis"

# Fetch product variants (combinations) — often missed:
curl -u "{API_KEY}:" "{site}/api/combinations?output_format=JSON&display=full&limit=20&filter[id_product]=[{N}]"

# Multi-store: route to specific store:
curl -u "{API_KEY}:" "{site}/api/products?output_format=JSON&id_shop=2"
```

**Cart AJAX (no auth):**
```bash
# Add product to cart — controller/action in POST body, not URL:
curl -s -X POST {site}/index.php \
  -b "PrestaShop-{hash}=...; PHPSESSID=..." \
  -d "add=1&action=update&id_product=1&qty=1&token={static_token}"

# Refresh cart fragments:
curl -s -X POST "{site}/index.php?fc=module&module=ps_shoppingcart&action=refresh"
```

**Hummingbird theme detection:** If `/themes/hummingbird/` appears in HTML src paths, the store
runs Hummingbird — do not rely on `window.prestashop` JS global; use cookie fingerprinting and
`/api/` endpoint probing for data extraction.

## 6. Source Map Patterns

PrestaShop Classic theme JS does **not** include source maps in production.
Hummingbird theme and custom webpack/Vite themes may expose maps.

```bash
# Check Hummingbird bundles for source maps:
curl -I {site}/js/chunk-vendors.{hash}.js.map
curl -I {site}/js/theme.{hash}.js.map

# Check for sourceMappingURL comment in Classic theme:
curl -s {site}/themes/classic/assets/js/theme.js | grep "sourceMappingURL"

# Module JS is most likely to have maps (custom-compiled modules):
curl -I {site}/modules/{module}/views/js/{script}.js.map
```

Retrieve the active theme name from `window.prestashop.page.theme_name` (Classic) or from
`/themes/` path patterns in HTML src attributes (both themes).

## 10. Gotchas

- **`outputMode=json` is DEPRECATED and removed in PS 8.** Use `?output_format=JSON` for all Web Services API requests. Sending `?outputMode=json` silently returns XML in PS 8+ — this is wrong and will cause JSON parse errors.

- **The Web Services API is disabled by default.** Most production stores have it turned off. A 404 on `/api/` is the normal state. Only probe further if you get 200 or 401. Do not treat 404 as a site fault.

- **`display=full` is required for complete field coverage.** Default responses return only a short list of resource IDs and xlink:href links. Append `&display=full` for all fields, or `&display=[field1,field2]` for specific fields. Skipping this means you get no product data — just a list of URLs.

- **`schema=synopsis` for field discovery.** Before fetching records, use `/api/products?schema=synopsis` to see all available field names and types without transferring data. This is faster and reveals custom fields added by modules.

- **Combinations API is separate from products.** Product variants (size, color, etc.) are not in `/api/products` — they are in `/api/combinations`. Each combination has a separate endpoint and is linked to products via `id_product`. Most reconnaissance tools miss variant enumeration entirely.

- **Hummingbird theme removes JS globals.** PS 8.1+ stores using the Hummingbird theme do not expose `window.prestashop`. Cart state, language, and currency data move to `localStorage` (browser-only). Use cookie fingerprinting (`PrestaShop-{hash}`) and the Web Services API instead of JS global inspection.

- **Controller URLs and friendly URLs coexist.** Use `/index.php?controller=` form for reliable probing — friendly URL patterns (`/{lang}/{category}/{product}.html`) vary by store configuration and language prefix.

- **The admin URL has a randomized suffix.** Back-office is at `/admin{hash}/` where the hash is generated at install time. Not discoverable via HTTP without source code or a directory scan.

- **Cart AJAX uses POST body parameters, not REST.** The controller, module, and action are in the POST body or URL query string. There is no `/api/cart` front-office REST endpoint — the Web Services `/api/orders` is back-office only.

- **`prestashop.cart` is live cart state (Classic theme).** The full cart is serialized into `window.prestashop.cart` on every page load — retrieve it without an API call by grepping the page source.

- **Language and currency switching is cookie-based.** The `id_lang` and `id_currency` cookies control active locale. Change these to probe the store in different languages or currencies. This affects product names, prices (if currency conversion is enabled), and URLs.

- **Module AJAX follows a consistent pattern.** All module AJAX routes through `/index.php?fc=module&module={name}&controller={controller}&ajax=1`. Enumerate by reading each module's JS files (in `/modules/{name}/views/js/`) for `$.ajax` or `fetch(` calls. Many modules also define routes via the `moduleRoutes` hook — these produce friendly URLs that are not discoverable from the source URL pattern.

- **Feed modules are the richest product data source for Greek-market stores.** Stores selling on Skroutz must maintain a Skroutz-format XML product feed. Look for `/modules/xmlfeedspro/`, `/modules/total_xml_exporter/`, or `/modules/skroutz_gr/export.php` — these expose the complete product catalog (name, SKU, price, weight, category, EAN) in XML format with no auth required. Similar logic applies to Google Shopping feeds.

- **PS 8.1+ Admin API is a completely separate system from the classic webservice.** It uses JWT auth, returns JSON-LD, lives at `/admin-api/` (not `/api/`), and requires the Admin API module installed. Do not conflate them. Check `/admin-api/docs.html` for the Swagger spec — it may be accessible without auth on misconfigured stores.

- **Multistore API requires `id_shop` parameter.** On multistore installations, `/api/products` defaults to the main store. Use `?id_shop=N` to target a specific store context, or enumerate all shops via `/api/shops`. Omitting `id_shop` may return 301/302 redirects to the store homepage on some PS 1.7/8 versions.

- **`/api/` root listing reveals installed module APIs.** The XML root response at `/api/` lists not just core PS resources but also any custom resources registered by installed modules (e.g., `ps_apiresources`). Always read the full resource list before assuming you know the API surface.
