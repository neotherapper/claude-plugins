# Shopware 6.x — Tech Pack

Shopware 6 is the dominant open-source e-commerce platform in the DACH market (Germany, Austria,
Switzerland), built on Symfony and PHP. It exposes three distinct API layers: a public-facing Store
API, a server-to-server Admin API (OAuth 2.0), and a bulk Sync API. Many European pen, stationery,
and specialty retail stores run Shopware 6, especially on `.de`, `.at`, and `.ch` domains. As of
April 2026 the current stable release is 6.7.9.0. Shopware's Composable Frontends (formerly Shopware
PWA) decouples the storefront into a Nuxt.js or Next.js front end that calls the Store API — on
headless deployments the API may be on a different subdomain than the rendered storefront.

## 1. Fingerprinting Signals

| Signal | Type | Value | Confidence |
|--------|------|-------|------------|
| `sw-context-token` cookie | Cookie | Name match | Definitive |
| `sw-currency` cookie | Cookie | Name match | High |
| `sw-language` cookie | Cookie | Name match | High |
| `sw-version` response header | HTTP header | Present in API responses | Definitive + version |
| `x-shopware-cache-id` response header | HTTP header | Semicolon-separated cache IDs | Definitive |
| `sw-storefront-url` response header | HTTP header | Storefront base URL | Definitive |
| `/store-api/` path in network requests | URL path | Substring match | Definitive |
| `window.__sw` or `window.sw` JS global | JS global | Object present in inline scripts | High |
| `shopware` in page source or meta generator | HTML substring | `<meta name="generator" content="Shopware">` | High |
| `/theme/{hash}/` paths in HTML | URL path | Compiled theme asset paths | High |
| `sw-access-key` in page source | HTML substring | `SWSC` prefix key in script or meta tag | High |
| `CSRFToken` meta tag or `sw-csrf` input field | HTML element | Name match | High |
| `meteor-icon-kit` or `shopware-icon` in HTML | HTML substring | Admin/plugin icon references | Medium |
| `.shopware.store` domain | Domain pattern | Shopware Cloud hosted stores | High |
| Wappalyzer detection | MCP | `"Shopware"` | Definitive |

**Version extraction:**
```bash
# From response headers (most reliable, no auth):
curl -sI {site} | grep -i "sw-version"

# From Store API info endpoint (public, no auth):
curl -s {site}/store-api/info \
  -H "sw-access-key: {key}" \
  -H "sw-context-token: {token}" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('swVersion',''))"

# From Admin API version endpoint (auth required, 6.3.5+):
curl -s {site}/api/_info/version \
  -H "Authorization: Bearer {token}"

# From Admin API config (auth required):
curl -s {site}/api/_info/config \
  -H "Authorization: Bearer {token}" \
  | grep -i "shopwareVersion"
```

## 2. Default API Surfaces

### 2a. Store API (public-facing, customer-side)

The Store API is Shopware 6's primary public API. All requests require the `sw-access-key` header.
Most stateful operations also require `sw-context-token`. The access key is always embedded in the
storefront page source in the format `SWSCXXXXXXXXXXXXXXXXXXXXXXXXXX`.

| Endpoint | Method | Auth | Notes |
|----------|--------|------|-------|
| `/store-api/context` | POST | Access key | Create anonymous session — returns `sw-context-token` |
| `/store-api/info` | GET | Access key | Shop name, currency, language, Shopware version |
| `/store-api/currency` | GET | Access key | Available currencies |
| `/store-api/language` | GET | Access key | Available languages with UUID IDs |
| `/store-api/country` | GET | Access key | Country list |
| `/store-api/category` | GET | Access key | Category tree |
| `/store-api/navigation/{activeId}/{rootId}` | GET | Access key | Navigation menu tree |
| `/store-api/product` | GET | Access key + token | Product listing |
| `/store-api/product/{productId}` | GET | Access key + token | Single product detail |
| `/store-api/product-listing/{categoryId}` | POST | Access key + token | Paginated products by category |
| `/store-api/search` | POST | Access key + token | Full product search |
| `/store-api/search-suggest` | POST | Access key + token | Search autocomplete (product names) |
| `/store-api/sitemap` | GET | Access key | Full site URL structure as JSON |
| `/store-api/cms-page/{id}` | POST | Access key + token | CMS page content blocks |
| `/store-api/cart` | GET | Access key + token | Cart contents |
| `/store-api/checkout/cart/line-item` | POST | Access key + token | Add item to cart |
| `/store-api/account/login` | POST | Access key + token | Customer login |
| `/store-api/customer` | GET | Access key + token | Current customer (login required) |

**Obtaining a context token (required for most Store API calls):**
```bash
# Extract the storefront access key from page source:
curl -s {site} | grep -oP '"accessKey"\s*:\s*"\K[^"]+' | head -1
# Alternative grep pattern:
curl -s {site} | grep -oP 'SWSC[A-Za-z0-9]+' | head -1

# Create anonymous context — always returns sw-context-token in response body and Set-Cookie:
curl -X POST {site}/store-api/context \
  -H "Content-Type: application/json" \
  -H "sw-access-key: {SWSC_KEY}" \
  -d "{}"

# Reuse the token on all subsequent Store API calls:
curl {site}/store-api/currency \
  -H "sw-access-key: {SWSC_KEY}" \
  -H "sw-context-token: {TOKEN}"
```

**Search autocomplete (no customer login required):**
```bash
curl -X POST "{site}/store-api/search-suggest" \
  -H "Content-Type: application/json" \
  -H "sw-access-key: {SWSC_KEY}" \
  -H "sw-context-token: {TOKEN}" \
  -d '{"search": "pen"}'
```

### 2b. Admin API (server-to-server, OAuth 2.0)

The Admin API is the full back-office API. All requests require a Bearer token obtained via OAuth.
On misconfigured dev or staging stores, the OpenAPI spec may be accessible without auth.

| Endpoint | Method | Auth | Notes |
|----------|--------|------|-------|
| `/api/oauth/token` | POST | Client credentials / password | Acquire Bearer token |
| `/api/_info/version` | GET | Bearer | Shopware version string (6.3.5+) |
| `/api/_info/config` | GET | Bearer | Version, enabled features, bundles |
| `/api/_info/openapi3.json` | GET | Bearer | Full OpenAPI spec for entire Admin API |
| `/api/_info/swagger.html` | GET | Bearer | Swagger UI (sometimes public on dev stores) |
| `/api/product` | GET | Bearer | Full product catalog with all fields |
| `/api/category` | GET | Bearer | Full category tree |
| `/api/order` | GET | Bearer | Order list |
| `/api/customer` | GET | Bearer | Customer list |
| `/api/currency` | GET | Bearer | Currency configuration |

**Admin API authentication:**
```bash
# Client credentials grant (integration-based):
curl -X POST {site}/api/oauth/token \
  -H "Content-Type: application/json" \
  -d '{"grant_type":"client_credentials","client_id":"{ID}","client_secret":"{SECRET}"}'

# Password grant (admin user):
curl -X POST {site}/api/oauth/token \
  -H "Content-Type: application/json" \
  -d '{"grant_type":"password","username":"admin","password":"{PASS}","client_id":"administration","scopes":"write"}'

# Use Bearer token:
curl {site}/api/product \
  -H "Authorization: Bearer {ACCESS_TOKEN}"
```

### 2c. Sync API (bulk operations)

| Endpoint | Method | Auth | Notes |
|----------|--------|------|-------|
| `/api/_action/sync` | POST | Bearer | Bulk create/update/delete for any entity |

### 2d. Headless / Composable Frontends

Shopware Composable Frontends (Nuxt.js or Next.js) consumes the Store API exclusively. On headless
deployments the Store API is often at a separate subdomain (e.g., `api.store.com`) while the
rendered storefront is at `www.store.com`. Detect by looking for Nuxt.js or Next.js signals in the
HTML alongside `/store-api/` calls in network traffic, or the presence of `@shopware-pwa` or
`@shopware-composables` in JS bundles.

## 3. Config / Constants Locations

| Location | What's there | How to access |
|----------|-------------|---------------|
| `window.__sw` or `window.sw` inline JS | Access key, languageId, currencyId, contextToken, theme config | JS eval / grep page source |
| `sw-access-key` in script tag or meta | Storefront sales channel key (`SWSC...`) | Grep page source for `SWSC` |
| `/store-api/info` | Shop name, currency, language config, `swVersion` | GET with access key (public) |
| `/store-api/context` POST response | Full context object: currency, language, customer group | POST with access key |
| `sw-context-token` cookie | Session state token | Set-Cookie header on first context POST |
| `/api/_info/config` | Shopware version, enabled bundles, active flags | GET with Admin Bearer token |
| `/api/_info/version` | Shopware version string only | GET with Admin Bearer token (6.3.5+) |
| `<meta name="generator">` | `Shopware` (may include version) | HTML scrape |
| `/theme/{hash}/js/all.js` | Compiled storefront JS — contains SW config object | Read from page source for hash |

## 4. Auth Patterns

| Pattern | Location | Notes |
|---------|----------|-------|
| `sw-access-key` header | Every Store API request | Storefront sales channel key; format `SWSCXXXXXXXXXX`; extracted from page source |
| `sw-context-token` header | Store API stateful requests | Anonymous session token; obtain via `POST /store-api/context`; tracks cart + customer |
| `sw-context-token` cookie | Browser sessions | Same token as header; set automatically by Shopware on first page load |
| OAuth 2.0 Bearer token | Admin API | Acquired via `POST /api/oauth/token`; client_credentials or password grant |
| `sw-language-id` header | Store API | UUID of the language; override active language per-request |
| `sw-currency-id` header | Store API | UUID of the currency; override active currency per-request |

**Storefront access key extraction (before any API call):**
```bash
# Method 1 — accessKey in JS config object:
curl -s {site} | grep -oP '"accessKey"\s*:\s*"\K[^"]+' | head -1

# Method 2 — raw SWSC pattern:
curl -s {site} | grep -oP 'SWSC[A-Za-z0-9]+' | head -1

# Method 3 — meta tag (some themes):
curl -s {site} | grep -oP '<meta name="sw-access-key"[^>]*content="\K[^"]+' | head -1
```

## 5. JS Bundle Patterns

| Path | Content |
|------|---------|
| `/theme/{themeHash}/js/all.js` | Compiled Storefront JS bundle — contains SW config, access key, plugin init |
| `/theme/{themeHash}/css/all.css` | Compiled theme CSS |
| `/bundles/storefront/storefront.js` | Core Storefront plugin JavaScript |
| `/bundles/{plugin-name}/administration/js/{plugin-name}.js` | Plugin admin panel JS |
| `/bundles/{plugin-name}/storefront/js/{plugin-name}.js` | Plugin storefront JS |
| `/bundles/molliepaymentsshopware6/` | Mollie Payments plugin bundle |
| `/bundles/rubon/` | Rubon theme plugin bundle |

The `{themeHash}` in `/theme/{hash}/` changes on every theme compilation. Always extract it fresh
from the page source HTML — never hardcode it. Look for `<link rel="stylesheet" href="/theme/` or
`<script src="/theme/` in the rendered HTML.

## 6. Source Map Patterns

Shopware 6 themes are compiled with webpack. Source maps are stripped in production by default.

```bash
# Check for source map on compiled JS bundle (usually 404 in production):
curl -I {site}/theme/{hash}/js/all.js.map

# Check for sourceMappingURL comment inside the bundle:
curl -s {site}/theme/{hash}/js/all.js | grep "sourceMappingURL"

# Plugin bundles may expose maps (custom-compiled plugins more likely):
curl -I {site}/bundles/{plugin-name}/storefront/js/{plugin-name}.js.map

# Composable Frontends (Nuxt/Next): check for _nuxt/ or /_next/ source maps:
curl -I {site}/_nuxt/entry.js.map
curl -I {site}/_next/static/chunks/main.js.map
```

If a map file returns 200, recover original TypeScript/Vue/JSX source paths from the `sources`
array in the map JSON — these reveal vendor libraries, plugin names, and internal module structure.

| Plugin | API / endpoint it adds | Detection signal |
|--------|----------------------|-----------------|
| Findologic (search) | Overrides `/store-api/search-suggest` — response format differs from native SW | Different JSON structure in search-suggest response; `findologic` in JS bundle |
| Mollie Payments | Payment flow JS; no new Store API endpoints | `/bundles/molliepaymentsshopware6/` path in HTML |

## 9. Probe Checklist

Run these in order. Record result (✓ 200 / ✗ 403 / – 404) for each.

- [ ] `HEAD {site}` — check for `sw-version`, `x-shopware-cache-id`, `sw-storefront-url` response headers (definitive fingerprints)
- [ ] `GET {site}` — grep for `sw-context-token` cookie, `window.__sw`, `window.sw`, `/theme/{hash}/`, `shopware` in meta generator, `SWSC` prefix
- [ ] Extract storefront access key from page source (`SWSC...` format) — required for all Store API calls
- [ ] Extract `{themeHash}` from `/theme/{hash}/` path in HTML — needed for JS bundle probing
- [ ] `POST {site}/store-api/context` with `sw-access-key` header — obtain `sw-context-token`; confirm 200 (412 = wrong/missing access key)
- [ ] `GET {site}/store-api/currency` — available currencies (confirms Store API active)
- [ ] `GET {site}/store-api/language` — all languages with UUID IDs (capture IDs for filtered queries)
- [ ] `GET {site}/store-api/info` — shop name, `swVersion`, currency and language config
- [ ] `GET {site}/store-api/category` — full category tree; capture category IDs for product listing
- [ ] `GET {site}/store-api/sitemap` — full JSON site structure
- [ ] `GET {site}/sitemap.xml` — XML sitemap fallback
- [ ] `POST {site}/store-api/search-suggest` with `{"search":"pen"}` — search autocomplete; confirm product names returned
- [ ] `POST {site}/store-api/product-listing/{categoryId}` — paginated product catalog with prices (use a category ID from step above)
- [ ] `GET {site}/store-api/customer` (with context token) — confirm customer endpoint exists (expect 403 if not logged in)
- [ ] `GET {site}/api/_info/config` — expect 401; confirms Admin API endpoint is live
- [ ] `GET {site}/api/_info/version` — expect 401 or version string if public
- [ ] `GET {site}/api/_info/openapi3.json` — expect 401; if 200, full Admin API spec is exposed
- [ ] `POST {site}/api/oauth/token` — test token endpoint exists (confirm 400 bad request, not 404)
- [ ] `GET {site}/theme/{hash}/js/all.js` — confirm bundle is accessible; check for `sourceMappingURL`
- [ ] `GET {site}/theme/{hash}/js/all.js.map` — check if source map exposed (usually 404)
- [ ] Check `/bundles/` paths in HTML for installed plugins: `molliepaymentsshopware6`, `rubon`, `frosh`, PayPal script tag
- [ ] `GET {site}/api/frosh-tools/` — check for Frosh Tools plugin (expect 401 or 404)
- [ ] `POST {site}/store-api/search-suggest` — compare response format to native SW; different structure = Findologic overriding search

## 10. Gotchas

- **Every Store API request requires `sw-access-key`.** Without it, Shopware returns HTTP 412 "No matching sales channel found" — not 401 or 403. This is the most common failure when starting Store API exploration. The key is always embedded in the storefront page source; find it with `grep -oP 'SWSC[A-Za-z0-9]+'`.

- **The context token (`sw-context-token`) is stateful session state.** It tracks cart contents, logged-in customer, selected language, and selected currency. Create one via `POST /store-api/context` and reuse it throughout the session. Dropping it mid-session results in a new anonymous context and lost cart state.

- **Product listing requires a `categoryId` — it cannot be listed without one.** Always enumerate categories first via `GET /store-api/category` or `GET /store-api/navigation`, then use a category ID in `POST /store-api/product-listing/{categoryId}`. The `/store-api/product` endpoint exists but returns limited results without proper filtering.

- **Search and listing use POST, not GET.** `POST /store-api/product-listing/{categoryId}` and `POST /store-api/search` both use POST with a JSON body for filters, sorting, and pagination. A GET request to these endpoints returns 405 or an empty result.

- **Language IDs are UUIDs, not ISO language codes.** The `sw-language-id` header and language filters require the UUID of the language (e.g., `2fbb5fe2e29a4d70aa5854ce7ce3e20b`), not `de`, `en`, or `de-DE`. Always fetch language UUIDs from `GET /store-api/language` before attempting language-filtered queries.

- **The Admin API OpenAPI spec at `/api/_info/openapi3.json` is extremely valuable but requires auth.** On misconfigured dev or staging stores it may be publicly accessible (200 response without Bearer token). Always probe it — a 200 here gives the complete Admin API surface map.

- **Shopware 6 is dominant in DACH but rare outside Europe.** Expect it on `.de`, `.at`, `.ch` domains where other platforms are less common. If Wappalyzer or header signals are ambiguous, the DACH domain pattern is a strong secondary signal.

- **Headless Shopware (Composable Frontends) separates the API from the storefront.** The Store API may be on `api.store.com` while the rendered front end is at `www.store.com`. In this case, the storefront HTML will show Nuxt.js or Next.js signals — look for XHR requests to `/store-api/` on a different origin to find the API host.

- **Plugin bundles at `/bundles/{plugin-name}/` reveal installed extensions.** Check each plugin path for storefront JS that contains AJAX endpoint definitions (`fetch(`, `axios.get(`). These reveal custom API endpoints added by the plugin that are not in the Store API or Admin API specs.

- **The `{themeHash}` in `/theme/{hash}/js/all.js` changes on every theme recompilation.** Never hardcode it. Extract it fresh from the rendered HTML `<script src="/theme/` or `<link href="/theme/` attributes. Caching the hash from a previous session will result in 404s after the store owner recompiles the theme.

- **`sw-access-key` key format is `SWSC` + alphanumeric characters.** If you can't find it with a SWSC grep, also check `<meta name="sw-access-key" content="...">` tags, the `window.__sw.accessKey` JS object property, and inline `<script>` blocks for `"accessKey":"SWSC..."` JSON patterns. Some third-party themes embed it differently.

- **On headless Shopware deployments, `SWSC` may not appear in page source.** If the frontend is a Nuxt/Next.js composable storefront, the access key may only be in the API server's environment — not exposed client-side. Fallback: `GET {site}/suggest?search=pen` is fully public and requires no key. It returns product names and URLs and works as an enumeration baseline even when the Store API is not directly accessible. Log `[SHOPWARE-HEADLESS:swsc-unavailable]` and use `/suggest` as the primary product data source.

- **Cart state is entirely tied to `sw-context-token`.** Every Store API call from a customer session must include this token as a header (`sw-context-token: {token}`) to maintain cart and customer state. The token is also set as a cookie by the Storefront, but for API calls you must send it explicitly as a header — cookie-only does not work for non-browser clients.
